## 1. Kubernetes 네트워킹 모델 개요

### 한 문장 정의

> Kubernetes 네트워킹은 **"모든 Pod가 NAT 없이 서로 직접 통신할 수 있어야 한다"**는 단순하지만 강력한 원칙 위에 설계되어 있습니다.

### 비유로 이해하기

```
┌─────────────────────────────────────────────────────────────┐
│                  회사 내선 전화 시스템 비유                      │
│                                                             │
│   기존 방식 (NAT 기반):                                       │
│   ┌────────┐    ┌──────────┐    ┌────────┐                   │
│   │ 직원 A │──▶│  교환원   │──▶│ 직원 B │                   │
│   │ 내선101│    │ (NAT)    │    │ 내선201│                   │
│   └────────┘    └──────────┘    └────────┘                   │
│   → 교환원이 모든 통화를 중계, 복잡하고 느림                     │
│                                                             │
│   Kubernetes 방식 (Flat Network):                             │
│   ┌────────┐                    ┌────────┐                   │
│   │ Pod A  │───── 직접 통화 ─────│ Pod B  │                   │
│   │10.244. │                    │10.244. │                   │
│   │  1.5   │                    │  2.10  │                   │
│   └────────┘                    └────────┘                   │
│   → 모든 Pod가 고유 IP로 직접 통신, 단순하고 빠름                │
└─────────────────────────────────────────────────────────────┘
```

### Kubernetes 네트워킹 4가지 기본 요구사항

```
┌─────────────────────────────────────────────────────────────┐
│         K8s 네트워크 모델의 4가지 필수 요구사항                  │
│                                                             │
│  ① 컨테이너 ↔ 컨테이너 (같은 Pod 내)                          │
│     └── localhost로 통신 (같은 네트워크 네임스페이스 공유)       │
│                                                             │
│  ② Pod ↔ Pod (노드 상관없이)                                  │
│     └── NAT 없이 고유 IP로 직접 통신                           │
│                                                             │
│  ③ Pod ↔ Service                                             │
│     └── Service의 가상 IP(ClusterIP)로 Pod 그룹에 접근         │
│                                                             │
│  ④ 외부 ↔ Service                                            │
│     └── NodePort, LoadBalancer, Ingress로 외부 트래픽 유입     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. CNI (Container Network Interface)

### CNI가 왜 필요한가?

> Kubernetes 자체는 **네트워크 구현을 포함하지 않습니다.** "모든 Pod가 서로 통신 가능해야 한다"는 요구사항만 정의하고, 실제 구현은 **CNI 플러그인**에 위임합니다.

```
┌─────────────────────────────────────────────────────────────┐
│              CNI가 없으면 어떤 일이 벌어질까?                   │
│                                                             │
│  Worker Node 1                  Worker Node 2               │
│  ┌─────┐                        ┌─────┐                    │
│  │Pod-A│ ─── 10.244.1.5 ───?──▶ │Pod-B│ 10.244.2.10       │
│  └─────┘                        └─────┘                    │
│                                                             │
│  ❌ Pod-A는 Pod-B의 IP를 알아도 패킷을 보낼 방법이 없음!       │
│  ❌ 노드 간 Pod 네트워크가 연결되어 있지 않음                   │
│  ❌ Pod에 IP를 할당할 방법도 없음                              │
│                                                             │
│  CNI가 하는 일:                                               │
│  ✅ Pod 생성 시 IP 주소 할당                                  │
│  ✅ Pod 네트워크 인터페이스(veth pair) 생성                     │
│  ✅ 노드 간 Pod 트래픽이 전달되도록 라우팅/터널링 설정           │
│  ✅ Pod 삭제 시 네트워크 자원 정리                              │
└─────────────────────────────────────────────────────────────┘
```

### CNI의 동작 원리

```
Pod 생성 요청
     │
     ▼
┌─── kubelet ──────────────────────────────────────────────┐
│                                                          │
│  1. Container Runtime에 Pod 샌드박스 생성 요청              │
│     └── 새 네트워크 네임스페이스 생성                        │
│                                                          │
│  2. CNI 플러그인 호출 (ADD 명령)                            │
│     └── /opt/cni/bin/<plugin> 바이너리 실행                │
│                                                          │
│  3. CNI 플러그인이 수행하는 작업:                            │
│     ├── veth pair 생성                                    │
│     │   (한쪽은 Pod 안, 다른 쪽은 호스트 네트워크)            │
│     ├── Pod에 IP 주소 할당 (IPAM)                          │
│     ├── 라우팅 테이블 설정                                  │
│     └── 결과를 JSON으로 kubelet에 반환                      │
│                                                          │
│  4. kubelet이 Pod 상태에 IP 정보 기록                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### veth pair란?

```
┌─────────────────────────────────────────────────────────────┐
│              veth pair (가상 이더넷 쌍)                        │
│                                                             │
│   "한쪽 끝에 데이터를 넣으면 다른 쪽 끝에서 나오는 파이프"       │
│                                                             │
│  ┌─── Pod 네트워크 NS ───┐    ┌─── 호스트 네트워크 ─────────┐│
│  │                       │    │                            ││
│  │  eth0 ◀══ veth pair ══▶ vethXXXX ──▶ 브릿지/라우터      ││
│  │  10.244.1.5           │    │              │             ││
│  │                       │    │              ▼             ││
│  └───────────────────────┘    │         다른 노드로 전달    ││
│                               │                            ││
│                               └────────────────────────────┘│
│                                                             │
│  Pod 안의 eth0 = veth pair의 한쪽 끝                          │
│  호스트의 vethXXXX = veth pair의 다른 쪽 끝                    │
│  → 이 연결을 통해 Pod 트래픽이 호스트 네트워크로 나감            │
└─────────────────────────────────────────────────────────────┘
```

### 주요 CNI 플러그인 비교

| CNI | 네트워크 방식 | 특징 | 적합한 환경 |
|-----|-------------|------|------------|
| **Calico** | BGP / IPIP / VXLAN | BGP 피어링으로 대규모 클러스터에 적합, 풍부한 NetworkPolicy | 온프레미스 대규모, 보안 중시 |
| **Cilium** | VXLAN / Geneve / Native | eBPF 기반 고성능, L7 정책, 커널 수준 처리 | 고성능 요구, 최신 커널 환경 |
| **Antrea** | VXLAN / Geneve / STT | VMware 공식, NSX 연동, OVS 기반 | vSphere/VMware 환경 |
| **Flannel** | VXLAN / host-gw | 단순한 구조, 학습용에 적합 | 소규모, 학습/테스트 |
| **Weave Net** | VXLAN (mesh) | 자동 메시 구성, 암호화 지원 | 소규모, 간편 설정 |

### 노드 간 Pod 통신 방식 비교

```
┌─── Overlay 방식 (VXLAN/Geneve) ──────────────────────────┐
│                                                          │
│  Pod-A (10.244.1.5)         Pod-B (10.244.2.10)         │
│     │                            ▲                      │
│     ▼                            │                      │
│  ┌────────┐   VXLAN 터널    ┌────────┐                   │
│  │ Node 1 │ ═══════════════ │ Node 2 │                   │
│  │192.168 │  원본 패킷을     │192.168 │                   │
│  │  .1.10 │  VXLAN으로 감쌈  │  .1.11 │                   │
│  └────────┘                 └────────┘                   │
│                                                          │
│  장점: 기존 네트워크 변경 없이 동작                          │
│  단점: 캡슐화/디캡슐화 오버헤드                              │
│                                                          │
├─── Native Routing 방식 (BGP/host-gw) ────────────────────┤
│                                                          │
│  Pod-A (10.244.1.5)         Pod-B (10.244.2.10)         │
│     │                            ▲                      │
│     ▼                            │                      │
│  ┌────────┐   IP 라우팅     ┌────────┐                   │
│  │ Node 1 │ ──────────────▶ │ Node 2 │                   │
│  │        │  BGP로 경로     │        │                   │
│  │        │  정보 교환       │        │                   │
│  └────────┘                 └────────┘                   │
│                                                          │
│  장점: 캡슐화 없어 성능 우수                                │
│  단점: 네트워크 인프라의 BGP 지원 필요                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### CNI 설정 확인

```bash
# CNI 바이너리 위치 확인
ls /opt/cni/bin/

# CNI 설정 파일 확인
ls /etc/cni/net.d/
cat /etc/cni/net.d/10-calico.conflist   # Calico 예시

# kubelet의 CNI 설정 확인
ps aux | grep kubelet | grep cni
# --cni-bin-dir=/opt/cni/bin
# --cni-conf-dir=/etc/cni/net.d

# 현재 사용 중인 CNI 확인
kubectl get pods -n kube-system | grep -E 'calico|cilium|antrea|flannel|weave'
```

## 3. Pod 네트워킹

### 같은 Pod 내 컨테이너 통신

```
┌─────────────────────────────────────────────────────────────┐
│          같은 Pod 내 컨테이너는 localhost로 통신                │
│                                                             │
│  ┌───────────── Pod (공유 네트워크 NS) ──────────────┐       │
│  │                                                   │       │
│  │  ┌──────────────┐      ┌──────────────┐           │       │
│  │  │ Container A  │      │ Container B  │           │       │
│  │  │ (nginx)      │      │ (log-agent)  │           │       │
│  │  │ :80          │      │ :8080        │           │       │
│  │  └──────┬───────┘      └──────┬───────┘           │       │
│  │         │                     │                   │       │
│  │         └──── localhost ──────┘                    │       │
│  │                                                   │       │
│  │  공유 자원:                                        │       │
│  │  ├── 네트워크 네임스페이스 (같은 IP, 같은 포트 공간)  │       │
│  │  ├── IPC 네임스페이스                               │       │
│  │  └── localhost (127.0.0.1)                         │       │
│  │                                                   │       │
│  │  Pod IP: 10.244.1.5                               │       │
│  │  eth0: ← CNI가 할당                                │       │
│  └───────────────────────────────────────────────────┘       │
│                                                             │
│  Container B에서 Container A 접근:                            │
│  curl http://localhost:80                                   │
│                                                             │
│  ⚠️ 같은 Pod 내에서는 포트 충돌 주의!                          │
│     두 컨테이너가 같은 포트를 사용할 수 없음                     │
└─────────────────────────────────────────────────────────────┘
```

### 같은 노드의 Pod 간 통신

```
┌─────── Worker Node 1 ────────────────────────────────────┐
│                                                          │
│  ┌─── Pod A ───┐         ┌─── Pod B ───┐                │
│  │ 10.244.1.5  │         │ 10.244.1.6  │                │
│  │    eth0     │         │    eth0     │                │
│  └──────┬──────┘         └──────┬──────┘                │
│         │ veth pair              │ veth pair              │
│    vethAAA                  vethBBB                      │
│         │                       │                        │
│         └───────┬───────────────┘                        │
│                 │                                        │
│          ┌──────┴──────┐                                 │
│          │   cbr0      │  ← 가상 브릿지                   │
│          │ (bridge)    │    (또는 CNI에 따라 다름)          │
│          └──────┬──────┘                                 │
│                 │                                        │
│          호스트 네트워크                                    │
│          192.168.1.10                                    │
└──────────────────────────────────────────────────────────┘

Pod A → Pod B 통신 경로:
1. Pod A의 eth0에서 패킷 발송 (dst: 10.244.1.6)
2. vethAAA를 통해 호스트 네트워크로 전달
3. 브릿지(cbr0)에서 목적지 MAC 주소 확인
4. vethBBB를 통해 Pod B로 전달
→ 같은 노드라서 매우 빠름!
```

### 다른 노드의 Pod 간 통신

```
┌─── Worker Node 1 ──────┐         ┌─── Worker Node 2 ──────┐
│                        │         │                        │
│  ┌─── Pod A ───┐       │         │  ┌─── Pod B ───┐       │
│  │ 10.244.1.5  │       │         │  │ 10.244.2.10 │       │
│  │    eth0     │       │         │  │    eth0     │       │
│  └──────┬──────┘       │         │  └──────┬──────┘       │
│         │              │         │         │              │
│    vethAAA             │         │    vethBBB             │
│         │              │         │         │              │
│  ┌──────┴──────┐       │         │  ┌──────┴──────┐       │
│  │   cbr0      │       │         │  │   cbr0      │       │
│  └──────┬──────┘       │         │  └──────┬──────┘       │
│         │              │         │         │              │
│    ┌────┴────┐         │         │    ┌────┴────┐         │
│    │ eth0    │         │         │    │ eth0    │         │
│    │192.168  │         │         │    │192.168  │         │
│    │ .1.10   │         │         │    │ .1.11   │         │
│    └────┬────┘         │         │    └────┬────┘         │
└─────────┼──────────────┘         └─────────┼──────────────┘
          │                                  │
          │    ┌──────────────────────┐       │
          └───▶│  물리 네트워크        │◀──────┘
               │  (VXLAN 터널 또는    │
               │   BGP 라우팅)        │
               └──────────────────────┘

Pod A → Pod B 통신 경로 (VXLAN 예시):
1. Pod A에서 패킷 발송 (src: 10.244.1.5, dst: 10.244.2.10)
2. Node 1의 cbr0에서 목적지가 로컬이 아님을 확인
3. CNI가 패킷을 VXLAN으로 캡슐화
   (외부: src=192.168.1.10, dst=192.168.1.11)
4. 물리 네트워크를 통해 Node 2로 전달
5. Node 2에서 VXLAN 디캡슐화
6. cbr0 → vethBBB → Pod B로 전달
```

## 4. Service와 kube-proxy

### Service가 필요한 이유

```
┌─────────────────────────────────────────────────────────────┐
```

### Service의 구성 요소

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-web-service       # 서비스 이름 (DNS에 등록됨)
spec:
  selector:
    app: my-web              # 이 라벨을 가진 Pod들을 대상으로 함
  ports:
  - port: 80                 # Service가 수신하는 포트
    targetPort: 8080         # Pod가 실제 리스닝하는 포트
    protocol: TCP
  type: ClusterIP            # 서비스 유형
```

```
┌─── Service ──────────────────────────────────────────────┐
│                                                          │
│  name: my-web-service                                    │
│  ClusterIP: 10.96.0.100                                  │
│                                                          │
│  selector: app=my-web                                    │
│      │                                                   │
│      │  라벨이 일치하는 Pod 자동 탐색                       │
│      │                                                   │
│      ├──▶ Pod-1 (app=my-web) → 10.244.1.5:8080          │
│      ├──▶ Pod-2 (app=my-web) → 10.244.1.6:8080          │
│      └──▶ Pod-3 (app=my-web) → 10.244.2.10:8080         │
│                                                          │

│              10.244.2.10:8080]                            │
```

### kube-proxy의 역할

> **kube-proxy**는 모든 노드에서 실행되는 네트워크 프록시로, Service의 가상 IP(ClusterIP)로 들어오는 트래픽을 **실제 Pod로 전달하는 규칙을 관리**합니다.

```
┌─────────────────────────────────────────────────────────────┐
│                kube-proxy 핵심 동작 원리                       │
│                                                             │
│  kube-proxy는 실제로 트래픽을 중계하지 않습니다!                │
│  대신 커널 수준의 네트워크 규칙을 설정합니다.                    │
│                                                             │
│  ┌─────────────────┐                                        │
│  │   API Server    │                                        │
│  │                 │  Service/Endpoints 변경 Watch           │
│  └────────┬────────┘                                        │
│           │                                                 │
│           ▼                                                 │
│  ┌─────────────────┐    iptables/IPVS 규칙 설정              │
│  │   kube-proxy    │ ──────────────────────────▶ 커널        │
│  │  (각 노드마다)   │                                        │
│  └─────────────────┘                                        │
│                                                             │
│  이후 트래픽 흐름 (kube-proxy 관여하지 않음):                   │
│                                                             │
│  Client Pod                                                 │
│     │                                                       │
│     │  dst: 10.96.0.100:80 (Service ClusterIP)              │
│     ▼                                                       │
│  ┌──────────────┐                                           │
│  │ 커널         │  iptables/IPVS 규칙 매칭                    │
│  │ (netfilter)  │  DNAT: 10.96.0.100 → 10.244.1.5           │
│  └──────┬───────┘                                           │
│         │                                                   │
│         ▼                                                   │
│  Pod (10.244.1.5:8080)   ← 실제 목적지로 직접 전달            │
│                                                             │
│  핵심: kube-proxy는 "규칙만 설정"하고, 실제 패킷 전달은         │
│        리눅스 커널(netfilter)이 수행!                          │
└─────────────────────────────────────────────────────────────┘
```

### kube-proxy 모드 비교

#### iptables 모드 (기본)

```
┌─── iptables 모드 ────────────────────────────────────────┐
│                                                          │
│  Service: my-web (10.96.0.100:80)                        │
│  Endpoints: Pod-1(10.244.1.5), Pod-2(10.244.1.6),        │
│             Pod-3(10.244.2.10)                           │
│                                                          │
│  생성되는 iptables 규칙 (단순화):                           │
│                                                          │
│  -A KUBE-SERVICES                                        │
│     -d 10.96.0.100/32 -p tcp --dport 80                  │
│     -j KUBE-SVC-XXXX                                     │
│                                                          │
│  -A KUBE-SVC-XXXX                                        │
│     -m statistic --mode random --probability 0.333       │
│     -j KUBE-SEP-AAA      ← 33% 확률로 Pod-1              │
│  -A KUBE-SVC-XXXX                                        │
│     -m statistic --mode random --probability 0.500       │
│     -j KUBE-SEP-BBB      ← 50% (남은 것 중) → 33% Pod-2  │
│  -A KUBE-SVC-XXXX                                        │
│     -j KUBE-SEP-CCC      ← 나머지 → 33% Pod-3            │
│                                                          │
│  -A KUBE-SEP-AAA                                         │
│     -j DNAT --to-destination 10.244.1.5:8080             │
│  -A KUBE-SEP-BBB                                         │
│     -j DNAT --to-destination 10.244.1.6:8080             │
│  -A KUBE-SEP-CCC                                         │
│     -j DNAT --to-destination 10.244.2.10:8080            │
│                                                          │
│  ✅ 안정적, 대부분의 환경에 적합                              │
│  ❌ Service/Pod 수가 많으면 규칙이 수천~수만 개로 증가         │
│  ❌ 규칙 탐색이 O(n) 선형 → 대규모에서 지연 증가              │
└──────────────────────────────────────────────────────────┘
```

#### IPVS 모드 (고성능)

```
┌─── IPVS 모드 ────────────────────────────────────────────┐
│                                                          │
│  IPVS (IP Virtual Server): 리눅스 커널의 L4 로드밸런서     │
│                                                          │
│  Service: my-web (10.96.0.100:80)                        │
│                                                          │
│  생성되는 IPVS 규칙:                                      │
│                                                          │
│  TCP  10.96.0.100:80 rr                                  │
│    -> 10.244.1.5:8080    Masq    1    0    0              │
│    -> 10.244.1.6:8080    Masq    1    0    0              │
│    -> 10.244.2.10:8080   Masq    1    0    0              │
│                                                          │
│  로드밸런싱 알고리즘 선택 가능:                              │
│  ├── rr    (Round Robin)       ← 기본값                   │
│  ├── lc    (Least Connection)  ← 연결 수 기반              │
│  ├── dh    (Destination Hash)  ← 목적지 기반 고정           │
│  ├── sh    (Source Hash)       ← 출발지 기반 고정           │
│  └── sed   (Shortest Expected Delay)                     │
│                                                          │
│  ✅ 해시 테이블 기반 → O(1) 탐색, 대규모에서 우수            │
│  ✅ 다양한 로드밸런싱 알고리즘 지원                           │
│  ❌ 커널에 IPVS 모듈 필요 (ip_vs, ip_vs_rr 등)            │
└──────────────────────────────────────────────────────────┘
```

#### 모드 비교 요약

| 항목 | iptables | IPVS |
|------|----------|------|
| **성능** | O(n) 규칙 탐색 | O(1) 해시 탐색 |
| **알고리즘** | 랜덤 확률 기반만 | rr, lc, dh, sh 등 선택 가능 |
| **적합 규모** | Service 1,000개 이하 | Service 수천 개 이상 |
| **안정성** | 매우 안정적 | 안정적 (커널 모듈 필요) |
| **기본값** | 대부분의 배포판 기본 | 명시적 설정 필요 |

## 5. CoreDNS

### CoreDNS가 필요한 이유

```
┌─────────────────────────────────────────────────────────────┐
│             IP 주소 대신 이름으로 통신하고 싶다!                 │
│                                                             │
│  ❌ IP 기반 접근의 문제:                                      │
│  curl http://10.96.0.100:80                                │
│  → Service IP가 변경되면? 모든 설정을 수정해야 함!              │
│  → IP 주소를 외울 수 있을까? 수십 개의 Service가 있다면?        │
│                                                             │
│  ✅ DNS 기반 접근 (CoreDNS):                                  │
│  curl http://my-web-service:80                              │
│  → 이름은 변하지 않음! Service가 재생성되어도 같은 이름          │
│  → 사람이 읽기 쉬움                                           │
│                                                             │
│  CoreDNS가 해결해주는 것:                                      │
│  "my-web-service" → 10.96.0.100  (DNS 조회)                  │
│  → Service 이름만 알면 어디서든 접근 가능!                      │
└─────────────────────────────────────────────────────────────┘
```

### ClusterIP (기본값)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-internal-svc
spec:
  type: ClusterIP              # 기본값 (생략 가능)
  selector:
    app: my-app

  - port: 80

```

```
┌─── 클러스터 내부 ───────────────────────────────────────────┐
│                                                            │
│  클러스터 내부에서만 접근 가능                                 │
│  ClusterIP: 10.96.0.100                                    │
│                                                            │
│  ┌─── Pod A ───┐                                           │
│  │ curl http://│──▶ 10.96.0.100:80 ──▶ Pod-1 (:8080)      │
```

## 7. LoadBalancer 서비스

### LoadBalancer가 필요한 이유

```
┌─────────────────────────────────────────────────────────────┐
│           NodePort만으로는 부족한 이유                          │
│                                                             │
│  NodePort의 한계:                                             │
│                                                             │
│  ❌ 사용자가 노드 IP를 직접 알아야 함                           │
│     http://192.168.1.10:30080                               │
│                                                             │
│  ❌ 특정 노드가 다운되면 접속 불가                              │
│     Node 1 장애 → http://192.168.1.10:30080 ✕               │
│                                                             │
│  ❌ 포트 번호가 비표준 (30000-32767)                           │
│     사용자에게 :30080 을 기억하라고?                            │
│                                                             │
│  ❌ 노드 간 트래픽 분산 기능 없음                               │
│     특정 노드에 트래픽이 몰릴 수 있음                           │
│                                                             │
│  ✅ LoadBalancer가 해결:                                      │
│     ├── 단일 외부 IP/도메인 제공                               │
│     ├── 표준 포트 사용 가능 (80, 443)                          │
│     ├── 노드 장애 시 자동 우회                                 │
│     └── 트래픽 자동 분산                                      │
└─────────────────────────────────────────────────────────────┘
```

### LoadBalancer 동작 원리

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-web
  annotations:
    # 클라우드별 추가 설정 (예: AWS)
    service.beta.kubernetes.io/aws-load-balancer-type: nlb
spec:
  type: LoadBalancer
  selector:
    app: my-web
  ports:
  - port: 80               # LB가 수신하는 포트
    targetPort: 8080        # Pod가 리스닝하는 포트
```

```
┌─────────────────────────────────────────────────────────────┐
│              LoadBalancer 트래픽 흐름                          │
│                                                             │
│  사용자 (인터넷)                                               │
│     │                                                       │
│     │  http://203.0.113.50:80  (외부 IP, 표준 포트)           │
│     ▼                                                       │
│  ┌──────────────────────────┐                                │
│  │   Cloud Load Balancer    │  ← 클라우드 제공자가 생성        │
│  │   (AWS ELB/NLB,         │     cloud-controller-manager    │
│  │    GCP GLB, Azure LB)   │     가 자동으로 프로비저닝        │
│  │                          │                                │
│  │  External IP: 203.0.113.50                                │
│  │  Health Check: 각 노드의 NodePort 체크                     │
│  └────────────┬─────────────┘                                │
│               │                                              │
│       ┌───────┴───────┐                                      │
│       │               │                                      │
│       ▼               ▼                                      │
│  ┌─ Node 1 ──┐  ┌─ Node 2 ──┐                               │
│  │  :30080   │  │  :30080   │   ← NodePort (자동 할당)       │
│  │     │     │  │     │     │                                │
│  │     ▼     │  │     ▼     │                                │
│  │ ClusterIP │  │ ClusterIP │   ← 내부 VIP                   │
│  │     │     │  │     │     │                                │
│  │  ┌──┴──┐  │  │  ┌──┴──┐  │                                │
│  │  │Pod-1│  │  │  │Pod-3│  │                                │
│  │  │Pod-2│  │  │  │Pod-4│  │                                │
│  │  └─────┘  │  │  └─────┘  │                                │
│  └───────────┘  └───────────┘                                │
│                                                             │
│  계층 구조:                                                   │
│  LoadBalancer = Cloud LB + NodePort + ClusterIP              │
│  (상위 유형은 하위 유형의 기능을 모두 포함)                      │
└─────────────────────────────────────────────────────────────┘
```

### 클라우드별 LoadBalancer 비교

| 클라우드 | LB 종류 | 특징 |
|---------|---------|------|
| **AWS** | CLB (Classic), NLB (Network), ALB (Application) | NLB는 L4, ALB는 L7 (Ingress Controller와 연동) |
| **GCP** | Network LB, HTTP(S) LB | HTTP(S) LB는 글로벌 L7 |
| **Azure** | Azure Load Balancer, Application Gateway | Standard/Basic SKU 선택 |
| **온프레미스** | MetalLB, kube-vip, OpenELB | 클라우드 LB 없이 LoadBalancer 타입 사용 가능 |

### 온프레미스에서 LoadBalancer 사용하기 (MetalLB)

```
┌─────────────────────────────────────────────────────────────┐
│  온프레미스 환경의 문제:                                       │
│  클라우드 LB가 없으므로 LoadBalancer 타입 Service를 생성하면    │
│  External IP가 영원히 <pending> 상태!                         │
│                                                             │
│  $ kubectl get svc                                          │
│  NAME     TYPE           CLUSTER-IP     EXTERNAL-IP         │
│  my-web   LoadBalancer   10.96.0.100    <pending>     ← !!  │
│                                                             │
│  해결: MetalLB 설치                                          │
│                                                             │
│  MetalLB 동작 방식:                                           │
│                                                             │
│  L2 모드 (Layer 2):                                          │
│  ├── 하나의 노드가 VIP에 대해 ARP 응답                         │
│  ├── 해당 노드로 모든 트래픽 집중 → kube-proxy가 분산           │
│  ├── 설정 간단, 네트워크 장비 변경 불필요                       │
│  └── 단점: 단일 노드 병목, 노드 장애 시 failover 지연          │
│                                                             │
│  BGP 모드:                                                   │
│  ├── 각 노드가 BGP로 VIP 경로를 라우터에 광고                  │
│  ├── 라우터가 ECMP로 트래픽을 여러 노드에 분산                  │
│  ├── 진정한 로드밸런싱 가능                                    │
│  └── 단점: 네트워크 장비의 BGP 지원 필요                       │
└─────────────────────────────────────────────────────────────┘
```

### externalTrafficPolicy

```yaml
apiVersion: v1

  name: my-web
spec:
  type: LoadBalancer
  externalTrafficPolicy: Local    # 또는 Cluster (기본값)

    app: my-web

    targetPort: 8080
```

```
┌─── externalTrafficPolicy: Cluster (기본값) ──────────────┐
│                                                          │
│  LB → Node 1 (:30080) → kube-proxy → Pod (어떤 노드든)  │
│                                                          │
│  ✅ 트래픽이 모든 Pod에 균등 분산                           │
│  ❌ 다른 노드의 Pod로 전달 시 추가 홉(hop) 발생             │
│  ❌ SNAT으로 클라이언트 원본 IP 손실                        │
│                                                          │
├─── externalTrafficPolicy: Local ─────────────────────────┤
│                                                          │
│  LB → Node 1 (:30080) → 해당 노드의 Pod만 전달            │
│  (Pod가 없는 노드는 Health Check 실패 → LB에서 제외)       │
│                                                          │
│  ✅ 추가 홉 없음, 지연 시간 감소                            │
│  ✅ 클라이언트 원본 IP 보존                                 │
│  ❌ Pod가 없는 노드는 트래픽을 받지 않아 불균형 가능          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 8. Ingress

### Ingress가 필요한 이유

```
┌─────────────────────────────────────────────────────────────┐
│           LoadBalancer만으로는 부족한 이유                      │
│                                                             │
│  서비스가 10개라면? LoadBalancer 10개 = 외부 IP 10개 필요!     │
│                                                             │
│  ┌─────────────┐  External IP: 203.0.113.50  ← 비용 💸      │
│  │ Service A   │  (LB #1)                                   │
│  │ (shop)      │                                            │
│  └─────────────┘                                            │
│  ┌─────────────┐  External IP: 203.0.113.51  ← 비용 💸      │
│  │ Service B   │  (LB #2)                                   │
│  │ (api)       │                                            │
│  └─────────────┘                                            │
│  ┌─────────────┐  External IP: 203.0.113.52  ← 비용 💸      │
│  │ Service C   │  (LB #3)                                   │
│  │ (admin)     │                                            │
│  └─────────────┘                                            │
│  ... x 10 = 막대한 비용! 관리도 복잡!                          │
│                                                             │
│  ✅ Ingress 사용 시:                                          │
│  ┌──────────────────┐  External IP: 203.0.113.50 (1개!)     │
│  │    Ingress        │                                       │
│  │                   │  shop.example.com → Service A         │
│  │  (L7 라우터)      │  api.example.com  → Service B         │
│  │                   │  admin.example.com → Service C        │
│  └──────────────────┘                                       │
│                                                             │
│  LB 1개로 여러 서비스에 라우팅! 비용 절감 + 중앙 관리           │
└─────────────────────────────────────────────────────────────┘
```

### Ingress vs LoadBalancer vs NodePort 비교

| 항목 | NodePort | LoadBalancer | Ingress |
|------|----------|--------------|---------|
| **계층** | L4 (TCP/UDP) | L4 (TCP/UDP) | L7 (HTTP/HTTPS) |
| **외부 접근** | 노드IP:포트 | 외부 IP:표준포트 | 도메인/경로 기반 |
| **SSL 종료** | ❌ | 일부 지원 | ✅ 중앙 관리 |
| **도메인 라우팅** | ❌ | ❌ | ✅ |
| **경로 라우팅** | ❌ | ❌ | ✅ |
| **비용 (Service당)** | 무료 | LB 1개 비용 | 공유 (LB 1개) |
| **적합 용도** | 개발/테스트 | L4 트래픽, TCP 서비스 | HTTP 웹 서비스 |

### Ingress 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                  Ingress 전체 구조                             │
│                                                             │
│  Ingress = 리소스(규칙 정의) + Controller(규칙 실행)           │
│                                                             │
│  ┌──────────────────────┐   ┌──────────────────────┐        │
│  │   Ingress 리소스      │   │  Ingress Controller  │        │
│  │   (YAML로 정의)       │──▶│  (실제 동작하는 Pod)  │        │
│  │                      │   │                      │        │
│  │  "shop.com → svc-a"  │   │  Nginx, Traefik,     │        │
│  │  "api.com → svc-b"   │   │  HAProxy, Istio 등   │        │
│  └──────────────────────┘   └──────────┬───────────┘        │
│                                        │                    │
│  ※ Ingress Controller는 별도 설치 필요!                       │
│    Kubernetes에 기본 포함되어 있지 않음!                        │
│                                                             │
│  주요 Ingress Controller:                                    │
│  ├── Nginx Ingress Controller (가장 널리 사용)                │
│  ├── Traefik (자동 설정, Let's Encrypt 통합)                 │
│  ├── HAProxy Ingress                                        │
│  ├── Istio Gateway (서비스 메시 환경)                         │
│  ├── AWS ALB Ingress Controller                              │
│  └── Contour (Envoy 기반)                                   │
└─────────────────────────────────────────────────────────────┘
```

### Ingress 동작 흐름

```
외부 사용자
   │
   │  https://shop.example.com/products
   ▼
┌──────────────────────────────────────────────────────────┐
│  Cloud LB / NodePort (Ingress Controller 앞단)            │
└──────────────────┬───────────────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────────────┐
│              Ingress Controller (예: Nginx)               │

```

### Ingress YAML 예시

#### 호스트 기반 라우팅 (도메인별 분기)

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: multi-host-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx           # 사용할 Ingress Controller 지정
  tls:
  - hosts:
    - shop.example.com
    - api.example.com
    secretName: tls-secret          # TLS 인증서 (Secret)
  rules:
  # 규칙 1: shop.example.com → shop-service
  - host: shop.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: shop-service
            port:
              number: 80

  # 규칙 2: api.example.com → api-service
  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
```

#### 경로 기반 라우팅 (URL 경로별 분기)

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: path-based-ingress
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.example.com
    http:
      paths:
      # /shop/* → shop-service
      - path: /shop
        pathType: Prefix
        backend:
          service:
            name: shop-service
            port:
              number: 80

      # /api/* → api-service
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80

      # /admin/* → admin-service
      - path: /admin
        pathType: Prefix
        backend:
          service:
            name: admin-service
            port:
              number: 80
```

```
┌─── 경로 기반 라우팅 시각화 ──────────────────────────────┐
│                                                          │
│  myapp.example.com                                       │
│         │                                                │
│         ├── /shop/*    ──▶  shop-service (3 Pods)        │
│         │                                                │
│         ├── /api/*     ──▶  api-service (5 Pods)         │
│         │                                                │
│         ├── /admin/*   ──▶  admin-service (2 Pods)       │
│         │                                                │
│         └── /* (기타)  ──▶  default-backend (404)         │
│                                                          │
│  하나의 도메인으로 여러 마이크로서비스에 라우팅!               │
└──────────────────────────────────────────────────────────┘
```


```yaml
# cert-manager를 사용한 자동 인증서 발급
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: auto-tls-ingress
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod   # 자동 발급
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - myapp.example.com
    secretName: myapp-tls          # cert-manager가 자동 생성
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp
            port:
              number: 80
```

### Ingress 주요 어노테이션 (Nginx)

| 어노테이션 | 설명 | 예시 |
|-----------|------|------|
| `rewrite-target` | URL 경로 재작성 | `/api(/\|$)(.*)` → `/$2` |
| `ssl-redirect` | HTTP → HTTPS 리다이렉트 | `"true"` |
| `proxy-body-size` | 요청 본문 최대 크기 | `"50m"` |
| `proxy-read-timeout` | 백엔드 응답 대기 시간 | `"60"` |
| `use-regex` | 경로에 정규식 사용 | `"true"` |
| `affinity` | 세션 어피니티(스티키 세션) | `"cookie"` |
| `limit-rps` | 초당 요청 수 제한 | `"10"` |
| `cors-allow-origin` | CORS 허용 오리진 | `"*"` |

## 9. NetworkPolicy

### NetworkPolicy란?

> **NetworkPolicy**는 Pod 수준의 방화벽 규칙입니다. 기본적으로 Kubernetes의 모든 Pod는 서로 자유롭게 통신할 수 있지만, NetworkPolicy를 사용하면 특정 트래픽만 허용하도록 제한할 수 있습니다.

```
┌─────────────────────────────────────────────────────────────┐
│         NetworkPolicy 적용 전 vs 후                           │
│                                                             │
│  적용 전 (기본): 모든 Pod가 서로 통신 가능                     │
│  ┌─────┐  ←→  ┌─────┐  ←→  ┌─────┐                         │
│  │ web │      │ api │      │ db  │                         │
│  └─────┘      └─────┘      └─────┘                         │
│  ⚠️ db Pod에 아무나 접근 가능! 보안 위험!                      │
│                                                             │
│  적용 후: 허용된 트래픽만 통과                                 │
│  ┌─────┐  ──▶  ┌─────┐  ──▶  ┌─────┐                       │
│  │ web │       │ api │       │ db  │                       │
│  └─────┘       └─────┘       └─────┘                       │
│     ╳ ─────────────────────────▶ ╳                          │
│  web → db 직접 접근 차단! api만 db 접근 가능!                  │
└─────────────────────────────────────────────────────────────┘
```

### NetworkPolicy 예시

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-allow-api-only
  namespace: default
spec:
  # 이 정책이 적용될 Pod 선택
  podSelector:
    matchLabels:
      app: db

  # 정책 유형: Ingress(들어오는 트래픽) 제한
  policyTypes:
  - Ingress

  # 허용 규칙
  ingress:
  - from:
    # api 라벨을 가진 Pod에서만 접근 허용
    - podSelector:
        matchLabels:
          app: api
    ports:
    - protocol: TCP
      port: 3306
```

```
┌─── 위 정책의 효과 ──────────────────────────────────────┐
│                                                         │
│  ✅ app=api Pod → app=db Pod (TCP 3306)    허용          │
│  ❌ app=web Pod → app=db Pod               차단          │
│  ❌ 다른 모든 Pod → app=db Pod              차단          │
│                                                         │
│  ※ CNI가 NetworkPolicy를 지원해야 함                     │
│     지원: Calico, Cilium, Antrea, Weave Net             │
│     미지원: Flannel (단독 사용 시)                        │
└─────────────────────────────────────────────────────────┘
```

### 자주 사용하는 NetworkPolicy 패턴

```yaml
# 1. 모든 Ingress 트래픽 차단 (기본 거부 정책)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
spec:
  podSelector: {}           # 모든 Pod에 적용
  policyTypes:
  - Ingress
  # ingress 규칙 없음 = 모두 차단
---
# 2. 특정 Namespace에서만 접근 허용
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-frontend-ns
spec:
  podSelector:
    matchLabels:
      app: api
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: frontend
---
# 3. 특정 CIDR(외부 IP 대역)에서만 접근 허용
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-office
spec:
  podSelector:
    matchLabels:
      app: admin
  ingress:
  - from:
    - ipBlock:
        cidr: 10.0.0.0/8
        except:
        - 10.0.1.0/24        # 특정 서브넷 제외
```

## 10. 네트워크 트러블슈팅

### 단계별 진단 흐름

```
문제: "서비스에 접근이 안 됩니다!"
        │
        ▼
┌─── 1단계: Pod 상태 확인 ─────────────────────────────────┐
│  kubectl get pods -l app=my-app                         │
│  → Pod가 Running 상태인가? (CrashLoopBackOff면 앱 문제)  │
│  → Pod의 IP가 할당되어 있는가? (CNI 문제)                 │
└──────────────┬──────────────────────────────────────────┘
               │ OK
               ▼
┌─── 2단계: Pod 직접 접근 테스트 ──────────────────────────┐
│  kubectl exec debug-pod -- curl <Pod-IP>:8080           │
│  → Pod 자체가 응답하는가? (앱 또는 포트 문제)              │
└──────────────┬──────────────────────────────────────────┘
               │ OK
               ▼
┌─── 3단계: Service → Pod 연결 확인 ───────────────────────┐
│  kubectl get endpoints my-service                       │
│  → Endpoints에 Pod IP가 있는가?                          │
│  → 없다면: selector 라벨 불일치, Pod readiness 실패       │
└──────────────┬──────────────────────────────────────────┘
               │ OK
               ▼
┌─── 4단계: Service DNS 확인 ──────────────────────────────┐
│  kubectl exec debug-pod -- nslookup my-service          │
│  → DNS 해석이 되는가? (CoreDNS 문제)                     │
└──────────────┬──────────────────────────────────────────┘
               │ OK
               ▼
┌─── 5단계: Service 접근 테스트 ────────────────────────────┐
│  kubectl exec debug-pod -- curl my-service:80           │
│  → ClusterIP로 접근되는가? (kube-proxy/iptables 문제)    │
└──────────────┬──────────────────────────────────────────┘
               │ OK
               ▼
┌─── 6단계: 외부 접근 확인 ────────────────────────────────┐
│  curl http://<NodeIP>:<NodePort>                        │
│  curl http://<ExternalIP>                               │
│  → 외부에서 접근되는가? (방화벽, LB, Ingress 문제)        │
└──────────────────────────────────────────────────────────┘
```

### 유용한 디버깅 명령어 모음

```bash
# ─── Pod 네트워크 확인 ───
# Pod의 IP 주소 확인
kubectl get pods -o wide

# Pod 내부에서 네트워크 테스트
kubectl exec -it <pod> -- ping <target-ip>
kubectl exec -it <pod> -- curl -v http://<service>:<port>
kubectl exec -it <pod> -- wget -qO- http://<service>:<port>

# Pod의 네트워크 설정 확인
kubectl exec <pod> -- ip addr
kubectl exec <pod> -- ip route
kubectl exec <pod> -- cat /etc/resolv.conf

# ─── Service/Endpoints 확인 ───
# Service의 Endpoints 확인 (Pod IP 목록)
kubectl get endpoints <service-name>
kubectl describe service <service-name>

# ─── DNS 디버깅 ───
# DNS 전용 디버그 Pod 실행
kubectl run dnsutils --image=registry.k8s.io/e2e-test-images/jessie-dnsutils:1.3 \
  --rm -it --restart=Never -- nslookup <service-name>

# CoreDNS 로그 확인
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=50

# ─── kube-proxy 확인 ───
# iptables 규칙 확인 (노드에서)
sudo iptables -t nat -L KUBE-SERVICES -n
sudo iptables -t nat -L KUBE-SVC-XXXX -n    # 특정 서비스

# IPVS 규칙 확인 (IPVS 모드일 때)
sudo ipvsadm -Ln

# ─── CNI 확인 ───
# CNI Pod 상태 확인
kubectl get pods -n kube-system | grep -E 'calico|cilium|antrea|flannel'

# CNI 로그 확인
kubectl logs -n kube-system <cni-pod-name> --tail=50

# ─── Ingress 확인 ───
# Ingress 리소스 확인
kubectl get ingress
kubectl describe ingress <ingress-name>

# Ingress Controller 로그
kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx --tail=50
```

### 자주 발생하는 문제와 해결

| 증상 | 가능한 원인 | 확인 방법 | 해결 |
|------|------------|----------|------|
| Pod에 IP 미할당 | CNI 장애 | `kubectl get pods -o wide` | CNI Pod 재시작, 로그 확인 |
| Endpoints가 비어있음 | selector 불일치 | `kubectl get ep <svc>` | Service selector와 Pod 라벨 비교 |
| DNS 해석 실패 | CoreDNS 다운 | `kubectl get pods -n kube-system -l k8s-app=kube-dns` | CoreDNS Pod 재시작 |
| ClusterIP 접근 불가 | kube-proxy 장애 | `iptables -t nat -L` | kube-proxy Pod 재시작 |
| NodePort 접근 불가 | 방화벽 차단 | `telnet <nodeIP> <nodePort>` | 방화벽 규칙 확인 |
| Ingress 502/504 | 백엔드 Pod 장애 | `kubectl describe ingress` | Pod 상태, readiness 확인 |
| 노드 간 Pod 통신 불가 | CNI 터널 문제 | 다른 노드 Pod에 ping | CNI 로그, 노드 간 방화벽 확인 |

## 12. 정리 및 핵심 요약

### 한눈에 보는 K8s 네트워킹

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   외부 트래픽                                                    │
│      │                                                          │
│      ▼                                                          │
│   Ingress (L7: 도메인/경로 라우팅, TLS 종료)                      │
│      │                                                          │
│      ▼                                                          │
│   LoadBalancer (L4: 외부 IP, 표준 포트)                           │
│      │                                                          │
│      ▼                                                          │
│   NodePort (노드 포트 개방)                                      │
│      │                                                          │
│      ▼                                                          │
│   ClusterIP (내부 VIP) ◀── kube-proxy (iptables/IPVS 규칙 관리)  │
│      │                                                          │
│      ▼                                                          │
│   Pod ◀── CNI (IP 할당, 네트워크 연결) ◀── CoreDNS (이름 해석)    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 핵심 포인트 7가지

| # | 포인트 | 설명 |
|---|--------|------|
| 1 | **CNI는 Pod 네트워크의 기반** | IP 할당, veth pair, 노드 간 라우팅/터널링을 담당. 없으면 Pod 통신 불가 |
| 2 | **kube-proxy는 규칙만 설정** | 실제 패킷 전달은 커널(iptables/IPVS)이 수행. 대규모에서는 IPVS 모드 고려 |
| 3 | **CoreDNS로 이름 기반 통신** | Service 이름으로 접근 가능. 같은 NS는 이름만, 다른 NS는 `svc.ns` 형식 |
| 4 | **ClusterIP는 내부 전용** | 외부 노출이 필요하면 NodePort, LoadBalancer, Ingress 사용 |
| 5 | **LoadBalancer는 L4** | Service당 외부 IP 1개. 온프레미스에서는 MetalLB 필요 |
| 6 | **Ingress는 L7 라우터** | 도메인/경로 기반 라우팅, TLS 종료, LB 1개로 여러 서비스 처리 |
| 7 | **NetworkPolicy는 Pod 방화벽** | 기본은 모두 허용. 명시적으로 트래픽 제한 가능 (CNI 지원 필요) |

### 언제 무엇을 사용해야 하는가?

```
┌─────────────────────────────────────────────────────────────┐
│                    선택 가이드                                 │
│                                                             │
│  Q: 클러스터 내부 서비스 간 통신만 필요?                        │
│  A: → ClusterIP (기본값)                                     │
│                                                             │
│  Q: 개발/테스트 환경에서 간단히 외부 노출?                      │
│  A: → NodePort                                              │
│                                                             │
│  Q: 프로덕션 환경에서 안정적인 외부 접근?                       │
│  A: → LoadBalancer (L4) 또는 Ingress (L7)                    │
│                                                             │
│  Q: 하나의 IP로 여러 HTTP 서비스를 도메인/경로로 분기?           │
│  A: → Ingress                                               │
│                                                             │
│  Q: TCP/UDP 서비스를 외부에 노출? (DB, gRPC 등)               │
│  A: → LoadBalancer                                          │
│                                                             │
│  Q: 외부 서비스(RDS 등)를 K8s 이름으로 접근?                   │
│  A: → ExternalName                                          │
│                                                             │
│  Q: Pod 간 통신을 보안 정책으로 제한?                           │
│  A: → NetworkPolicy                                         │
└─────────────────────────────────────────────────────────────┘
```

### CKA/CKAD 시험 빈출 포인트

| 주제 | 자주 나오는 문제 유형 |
|------|---------------------|
| Service 생성 | `kubectl expose` 로 ClusterIP/NodePort 생성 |
| Ingress | 호스트/경로 기반 라우팅 YAML 작성 |
| NetworkPolicy | 특정 Pod 간 트래픽 허용/차단 정책 작성 |
| DNS 디버깅 | `nslookup`으로 Service 이름 해석 테스트 |
| CoreDNS | Corefile 수정, DNS 문제 해결 |
| Endpoints | Service와 Pod 연결 상태 확인 |
| kube-proxy | iptables 규칙 확인, 모드 이해 |
