# VCF 9.0 Supervisor — VPC 네트워킹 교육

> VMware Cloud Foundation(VCF) 9.0 환경에서 **Supervisor를 VPC(Virtual Private Cloud) 네트워킹**으로 배포하고 운영하는 방법을 다룹니다.
> VPC는 VCF Automation 환경의 **권장 워크로드 네트워크 스택**으로, NSX 프로젝트 기반의 멀티테넌시와 세분화된 네트워크 격리를 제공합니다.

## 1. VPC 네트워킹 개요

### 한 문장 정의

> **VPC(Virtual Private Cloud)**는 VCF 9.0에서 Supervisor 워크로드 네트워킹의 **권장 네트워크 스택**으로, NSX 프로젝트 내에 생성되어 테넌트별 격리된 네트워크 환경을 제공합니다.

### 왜 VPC인가?

```
┌─────────────────────────────────────────────────────────────────┐
│              기존 NSX-T vs VPC 네트워킹 비교                       │
│                                                                 │
│  ┌──────────────────────┐    ┌──────────────────────┐           │
│  │  기존 NSX-T 방식      │    │  VPC 방식 (VCF 9.0)  │           │
│  │                      │    │                      │           │
│  │  • Tier-0/Tier-1     │    │  • NSX Project 기반   │           │
│  │    게이트웨이 직접     │    │  • VPC 단위 격리      │           │
│  │    구성 필요          │    │  • 자동화된 네트워크    │           │
│  │  • 세그먼트 수동 생성  │    │    프로비저닝          │           │
│  │  • 복잡한 설정        │    │  • VCF Automation     │           │
│  │                      │    │    통합               │           │
│  └──────────────────────┘    └──────────────────────┘           │
│                                                                 │
│  VPC = VCF Automation의 권장 네트워크 스택                         │
│  → 멀티테넌시, 자동화, 간편한 운영에 최적화!                         │
└─────────────────────────────────────────────────────────────────┘
```

### 핵심 개념 관계도

```
┌─────────────────────────────────────────────────────────────────┐
│                    VPC 핵심 개념 관계                              │
│                                                                 │
│  VCF Automation Tenant (테넌트)                                  │
│    │                                                            │
│    ├── NSX Project A ──────────────────────────────┐            │
│    │     │                                         │            │
│    │     ├── VPC 1 ── vSphere Namespace 1          │            │
│    │     │     ├── Private Subnet                  │            │
│    │     │     ├── External Subnet                 │            │
│    │     │     └── Private TGW Subnet              │            │
│    │     │                                         │            │
│    │     └── VPC 2 ── vSphere Namespace 2          │            │
│    │           ├── Private Subnet                  │            │
│    │           ├── External Subnet                 │            │
│    │           └── Private TGW Subnet              │            │
│    │                                               │            │
│    └── NSX Project B ──────────────────────────────┘            │
│          └── VPC 3 ── vSphere Namespace 3                      │
│                                                                 │
│  ※ 1 Tenant = N개 Project, 1 Project = N개 VPC                  │
│  ※ 각 VPC는 독립적인 라우팅 도메인                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. Supervisor + VPC 아키텍처

### Centralized Transit Gateway 아키텍처

VPC에서 **Centralized Transit Gateway**를 사용하면 트래픽이 **NSX Edge 노드의 NSX 게이트웨이**를 통해 라우팅됩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│         Supervisor + VPC + Centralized Transit Gateway          │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    물리 네트워크                           │    │
│  │                  (BGP / Static Route)                    │    │
│  └───────────────────────┬─────────────────────────────────┘    │
│                          │                                      │
│                 ┌────────▼────────┐                              │
│                 │   Tier-0 GW     │ ← NSX Edge 클러스터          │
│                 │ (Active/Standby)│                              │
│                 └────────┬────────┘                              │
│                          │                                      │
│              ┌───────────▼───────────┐                          │
│              │   Transit Gateway     │                          │
│              │ (Centralized 연결)     │                          │
│              └───────────┬───────────┘                          │
│                          │                                      │
│         ┌────────────────┼────────────────┐                     │
│         │                │                │                     │
│    ┌────▼────┐     ┌─────▼────┐     ┌─────▼────┐              │
│    │  VPC 1  │     │  VPC 2   │     │System VPC│              │
│    │(NS: app)│     │(NS: web) │     │(Sup.Svc) │              │
│    │         │     │          │     │          │              │
│    │ Private │     │ Private  │     │ Private  │              │
│    │ External│     │ External │     │ External │              │
│    │ Priv.TGW│     │ Priv.TGW │     │ Priv.TGW │              │
│    └─────────┘     └──────────┘     └──────────┘              │
│                                                                 │
│    ※ NCP가 자동 생성하는 VPC:                                     │
│      • System Namespace용 VPC (NSX System Project)              │
│      • Supervisor Service용 VPC                                 │
│      • 각 vSphere Namespace별 개별 VPC                           │
└─────────────────────────────────────────────────────────────────┘
```

### NCP(NSX Container Plugin)가 자동 생성하는 리소스

| 자동 생성 VPC | 위치 | 용도 |
|--------------|------|------|
| **System Namespace VPC** | NSX System Project | Supervisor 시스템 네임스페이스 네트워킹 |
| **Supervisor Service VPC** | — | Supervisor 서비스 네트워킹 |
| **개별 Namespace VPC** | 지정된 NSX Project | 각 vSphere Namespace별 독립 네트워킹 |

## 3. VPC 서브넷 유형과 트래픽 흐름

### 세 가지 서브넷 유형

```
┌─────────────────────────────────────────────────────────────────┐
│                    VPC 서브넷 유형 비교                            │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  1. Private Subnet (기본값)                              │    │
│  │     ─────────────────────────────────                    │    │
│  │     • VPC 내부에서만 접근 가능                              │    │
│  │     • 워크로드 간 내부 통신에 사용                           │    │
│  │     • VM Service VM의 기본 배치 위치                        │    │
│  │     • vSphere Pod의 기본 SubnetSet                        │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  2. External Subnet (Public)                            │    │
│  │     ─────────────────────────────────                    │    │
│  │     • 프로젝트 내부 + 외부 모두 접근 가능                    │    │
│  │     • LoadBalancer Service / Ingress에 사용               │    │
│  │     • NSX Project의 Egress에 사용                         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  3. Private TGW (Transit Gateway) Subnet                │    │
│  │     ─────────────────────────────────                    │    │
│  │     • 같은 NSX Project 내 다른 VPC에서 접근 가능            │    │
│  │     • VPC Egress의 SNAT에 사용                            │    │
│  │     • vSphere Pod에 사용                                  │    │
│  │     • VPC 간 통신이 필요할 때 활용                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 트래픽 흐름 다이어그램

```
┌─────────────────────────────────────────────────────────────────┐
│                      트래픽 흐름 예시                              │
│                                                                 │
│  [외부 사용자]                                                    │
│       │                                                         │
│       ▼                                                         │
│  ┌──────────┐     ┌──────────────┐     ┌─────────────────┐     │
│  │ 물리 NW  │────▶│  Tier-0 GW   │────▶│ Transit GW      │     │
│  └──────────┘     └──────────────┘     └────────┬────────┘     │
│                                                  │              │
│                         ┌────────────────────────┼───────┐      │
│                         │        VPC             │       │      │
│                         │                        ▼       │      │
│                         │  ┌──────────────────────────┐  │      │
│                         │  │   External Subnet        │  │      │
│                         │  │  (LoadBalancer VIP)       │  │      │
│                         │  └───────────┬──────────────┘  │      │
│                         │              │                 │      │
│                         │              ▼                 │      │
│                         │  ┌──────────────────────────┐  │      │
│                         │  │   Private Subnet         │  │      │
│                         │  │  (워크로드 Pod / VM)       │  │      │
│                         │  └───────────┬──────────────┘  │      │
│                         │              │                 │      │
│                         │              ▼                 │      │
│                         │  ┌──────────────────────────┐  │      │
│                         │  │  Private TGW Subnet      │  │      │
│                         │  │  (VPC간 통신 / SNAT)      │  │      │
│                         │  └──────────────────────────┘  │      │
│                         └────────────────────────────────┘      │
│                                                                 │
│  ※ VM Service VM은 모든 서브넷 유형에 배치 가능                     │
│    (기본값: Private Subnet)                                      │
└─────────────────────────────────────────────────────────────────┘
```

### Kubernetes CRD로 노출되는 VPC/네트워크 리소스

> VPC와 네트워크 구성은 **Kubernetes CRD(Custom Resource Definition)**로 Supervisor API에 노출됩니다.
> Namespace 역할을 가진 사용자가 서브넷을 생성하고 관리할 수 있습니다.

| CRD | 설명 |
|-----|------|
| VPC | VPC 리소스 정의 |
| Subnet / SubnetSet | 서브넷 관리 (기본 SubnetSet은 자동 생성) |
| NetworkPolicy | K8s 네트워크 정책 |
| Service (LoadBalancer) | 로드밸런서 VIP 할당 |

## 4. 로드 밸런서 구성

### 로드 밸런서 선택

```
┌─────────────────────────────────────────────────────────────────┐
│                  로드 밸런서 옵션                                  │
│                                                                 │
│  옵션 1: NSX Load Balancer (기본)                                │
│  ┌──────────────────────────────────────────────┐               │
│  │  • NSX Edge 노드에서 실행                      │               │
│  │  • 별도 설치 불필요                             │               │
│  │  • Avi가 없으면 자동 선택                       │               │
│  └──────────────────────────────────────────────┘               │
│                                                                 │
│  옵션 2: Avi Load Balancer (선택)                                │
│  ┌──────────────────────────────────────────────┐               │
│  │  • 별도 설치/구성 필요                          │               │
│  │  • 고급 L7 기능 제공                            │               │
│  │  • AKO (Avi Kubernetes Operator) 사용          │               │
│  └──────────────────────────────────────────────┘               │
│                                                                 │
│  ⚠️  중요 제약사항:                                               │
│  ─────────────────────────────────────────────                  │
│  • Supervisor와 VPC는 반드시 같은 LB를 사용해야 함                  │
│  • Supervisor가 NSX LB → VPC도 NSX LB만 사용 가능                 │
│  • 두 가지를 혼합 사용할 수 없음!                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### NCP와 AKO의 역할

| 컴포넌트 | 역할 |
|---------|------|
| **NCP (NSX Container Plugin)** | PodVM 네트워킹, 서브넷 프로비저닝, IPAM, NetworkPolicy |
| **AKO (Avi Kubernetes Operator)** | ClusterIP, LoadBalancer, Ingress 서비스 타입 지원 |

## 5. 스토리지 정책 생성

### 스토리지 정책이 필요한 이유

```
┌─────────────────────────────────────────────────────────────────┐
│               스토리지 정책 적용 대상                               │
│                                                                 │
│  Supervisor 활성화 시:                                            │
│  ┌────────────────────────────────────────────────┐             │
│  │  • Supervisor Control Plane VM 배치              │             │
│  │  • vSphere Pod 임시 디스크                        │             │
│  │  • 컨테이너 이미지 저장                            │             │
│  └────────────────────────────────────────────────┘             │
│                                                                 │
│  Namespace 할당 시:                                               │
│  ┌────────────────────────────────────────────────┐             │
│  │  • Persistent Volume 할당                        │             │
│  │  • VM Content Library                            │             │
│  │  • VKS 클러스터 노드 배포                          │             │
│  │  → K8s StorageClass로 자동 매핑됨!                │             │
│  └────────────────────────────────────────────────┘             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 스토리지 정책 생성 절차

```
┌─────────────────────────────────────────────────────────────────┐
│              스토리지 정책 생성 흐름 (예: Gold)                     │
│                                                                 │
│  Step 1: 데이터스토어에 태그 지정                                  │
│  ──────────────────────────────                                 │
│  Datastore (Gold) → 우클릭 → Tags and Custom Attributes        │
│    → Assign Tag → Add Tag                                      │
│      Name: Gold                                                │
│      Category: "Storage for Kubernetes" (새로 생성 가능)          │
│                                                                 │
│  Step 2: VM 스토리지 정책 생성                                    │
│  ──────────────────────────────                                 │
│  Menu → Policies and Profiles → VM Storage Policies            │
│    → Create VM Storage Policy                                  │
│      Name: goldsp                                              │
│      K8s Compliant Name: goldsp (RFC 1123 호환)                 │
│                                                                 │
│  Step 3: 정책 구조 설정                                           │
│  ──────────────────────────────                                 │
│  Policy Structure 페이지:                                       │
│    ✅ Datastore specific rules → Tag-based placement 활성화      │
│    ✅ vSphere Pod + VMFS → Volume allocation: Fully initialized │
│    ✅ 3-Zone Supervisor → Enable Zonal topology 활성화           │
│                                                                 │
│  Step 4: 태그 기반 배치 규칙 설정                                  │
│  ──────────────────────────────                                 │
│  Tag based placement:                                          │
│    Category: Storage for Kubernetes                            │
│    Usage: Use storage tagged with                              │
│    Tag: Gold                                                   │
│                                                                 │
│  Step 5: 호환성 확인 → 완료                                       │
│  ──────────────────────────────                                 │
│  Storage compatibility → 정책에 맞는 데이터스토어 목록 확인         │
│  Review and Finish → Finish                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 스토리지 정책 유형별 가이드

| 구분 | 1-Zone Supervisor | 3-Zone Supervisor |
|------|------------------|------------------|
| **Topology Aware** | 불필요 | 필수 (Enable Zonal topology) |
| **Consumption Domain** | 비활성화 | 활성화 |
| **공유 데이터스토어** | 클러스터 내 모든 ESX 호스트 공유 | Zone별 독립 구성 |
| **지원 유형** | VMFS, NFS, vSAN | VMFS, NFS, vSAN |

### 필수 권한

- `VM storage policies > Update`
- `VM storage policies > View`

## 6. 배포 요구사항

### 최소 컴퓨팅 요구사항

```
┌─────────────────────────────────────────────────────────────────┐
│               관리 및 Edge 클러스터 최소 사양                       │
│                                                                 │
│  ┌─────────────────┬────────┬────────┬─────────┬──────────┐    │
│  │    시스템         │  규모   │  CPU   │ 메모리   │ 스토리지  │    │
│  ├─────────────────┼────────┼────────┼─────────┼──────────┤    │
│  │ vCenter 9.0     │ Small  │   2    │  21 GB  │  290 GB  │    │
│  │ NSX Manager     │ Medium │   6    │  24 GB  │  300 GB  │    │
│  │ NSX Edge (x2)   │ Large  │   8    │  32 GB  │  200 GB  │    │
│  │ K8s CP VM       │  -     │   4    │  16 GB  │   16 GB  │    │
│  └─────────────────┴────────┴────────┴─────────┴──────────┘    │
│                                                                 │
│  vSphere 클러스터:                                               │
│  • 1개 또는 3개 클러스터 (3개 = 3-Zone 구성)                       │
│  • vSphere DRS (Fully Automate) + HA 활성화 필수                 │
│  • 클러스터별 독립 스토리지/네트워킹                                │
│                                                                 │
│  ESX 호스트 9.0:                                                 │
│  • vSAN 미사용: 최소 1호스트 (권장 3)                              │
│  • vSAN 사용: 최소 3호스트 + 물리 NIC 2개 (권장 4)                 │
│  • 호스트 이름: 소문자만 사용!                                     │
│  • 호스트당: CPU 8코어, 메모리 64GB                               │
│                                                                 │
│  NSX Edge 클러스터:                                               │
│  • 2노드 클러스터, Large 폼팩터                                    │
│  • Active/Standby 모드                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 워크로드 네트워크 요구사항

| 구성 요소 | 최소 규모 | 설명 |
|----------|---------|------|
| **K8s Services CIDR** | /16 Private | K8s 서비스용 사설 CIDR. Supervisor마다 고유해야 함 |
| **Private IP Block (TGW)** | /16 Private | VPC 간 연결용 사설 CIDR. Tier-0에서 광고하지 않음 |
| **Private VPC CIDR** | /16 Private | Private 서브넷 IP 할당용. VPC 외부로 광고 안 됨. /26은 Avi SE 예약 |

### VPC 네트워킹 요구사항

| 구성 요소 | 최소 수량 | 필수 구성 |
|----------|---------|----------|
| **Centralized Connection** | 1 | 외부 연결 정의. Tier-0 게이트웨이 지정 |
| **Transit Gateway** | 1 | Centralized Connection 필요. Tier-0의 HA 모드 상속 (Active/Standby) |
| **NSX Project** | 1 | VPC 생성 위치. 기본 프로젝트가 아니면 별도 설정 필요 |
| **VPC Connectivity Profile** | 1 | Private-TGW IP Block, External IP Block, Auto SNAT 설정 |

> **주의:** External IP Block은 물리 네트워크의 VLAN/게이트웨이, vCenter의 VLAN 포트 그룹, NSX의 오버레이와 겹치면 안 됩니다!

## 10. Centralized Gateway 구성

### Centralized Gateway란?

```
┌─────────────────────────────────────────────────────────────────┐
│               Centralized Gateway 역할                           │
│                                                                 │
│  Centralized Gateway = VPC의 외부 연결 경로                       │
│                                                                 │
│  [외부 네트워크] ←→ [Tier-0 GW] ←→ [Transit GW] ←→ [VPC들]      │
│                                                                 │
│  • 모든 NSX 서비스에 대한 전체 접근이 필요할 때 사용                 │
│  • NSX Edge 노드를 통해 트래픽 라우팅                              │
│  • Workload Domain 생성 시 Supervisor를 활성화하면                 │
│    Centralized Gateway 구성 전까지                                │
│    vSphere Namespace를 생성할 수 없음                             │
│                                                                 │
│  구성 시 필요한 사항:                                              │
│  ┌──────────────────────────────────────────────┐               │
│  │  ✅ NSX Edge 노드 배포 완료                    │               │
│  │  ✅ Tier-0 게이트웨이 생성                      │               │
│  │  ✅ Centralized Connection 설정                │               │
│  │  ✅ Transit Gateway 구성                       │               │
│  │  ✅ VPC Connectivity Profile 생성              │               │
│  └──────────────────────────────────────────────┘               │
│                                                                 │
│  Transit Gateway:                                               │
│  • Centralized Connection과 함께 구성                             │
│  • Tier-0 GW의 HA 모드 상속 → Active/Standby 필수               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 12. 핵심 개념 비교 정리

### NSX Project vs VPC vs vSphere Namespace

```
┌─────────────────────────────────────────────────────────────────┐
│           핵심 개념 계층 구조 비교                                  │
│                                                                 │
│  ┌──────────────────────────────────────────────────┐           │
│  │  VCF Automation Tenant                           │           │
│  │  (테넌트 = 조직/팀 단위)                           │           │
│  │                                                  │           │
│  │  └── NSX Project                                 │           │
│  │      (테넌트의 네트워크 프로젝트)                   │           │
│  │      • 1 Tenant = N개 Project                    │           │
│  │      • Project = VPC들의 컨테이너                  │           │
│  │      • External IP Block, Edge 클러스터 지정       │           │
│  │                                                  │           │
│  │      └── VPC                                     │           │
│  │          (격리된 네트워크 도메인)                   │           │
│  │          • 독립적 라우팅 도메인                     │           │
│  │          • 3가지 서브넷 유형 포함                   │           │
│  │          • vSphere Namespace와 연결               │           │
│  │                                                  │           │
│  │          └── vSphere Namespace                   │           │
│  │              (K8s 네임스페이스)                    │           │
│  │              • VPC의 네트워크를 사용                │           │
│  │              • Pod, VM, VKS 클러스터 실행          │           │
│  │              • StorageClass, ResourceQuota 적용   │           │
│  └──────────────────────────────────────────────────┘           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Single Zone vs Three Zone 비교

| 구분 | Single Zone | Three Zones |
|------|-------------|-------------|
| **CP VM 수** | 1개 (HA 시 3개) | 3개 (Zone당 1개) |
| **장애 격리** | 클러스터 레벨 | Zone(클러스터) 레벨 |
| **최소 ESX** | 3개 | 9개 (Zone당 3개) |
| **스토리지 정책** | 일반 정책 | Topology Aware 필수 |
| **관리 네트워크** | 1개 | Zone별 개별 가능 |
| **권장 용도** | 테스트/PoC | 프로덕션 |

### Control Plane 크기별 비교

| 크기 | CPU | 메모리 | 스토리지 | 권장 용도 |
|------|-----|--------|---------|----------|
| **Tiny** | 2 | 8 GB | 32 GB | 테스트/PoC |
| **Small** | 4 | 16 GB | 32 GB | 소규모 환경 |
| **Medium** | 8 | 16 GB | 32 GB | 중규모 환경 |
| **Large** | 16 | 32 GB | 32 GB | 대규모 프로덕션 |

> **주의:** Scale Up만 가능하고 Scale Down은 불가하므로, 초기에 적절한 크기를 선택하는 것이 중요합니다.

## 13. 트러블슈팅 체크리스트

### 배포 전 확인사항

```
┌─────────────────────────────────────────────────────────────────┐
│               배포 전 체크리스트                                   │
│                                                                 │
│  네트워크:                                                       │
│  □ ESX Host VTEP + Edge VTEP MTU > 1600 확인                   │
│  □ Tier-0 Uplink IP가 고유한지 확인                               │
│  □ External IP Block이 기존 VLAN/오버레이와 겹치지 않는지 확인      │
│  □ 물리 네트워크에서 External IP Block에 접근 가능한지 확인          │
│  □ BGP 또는 Static Route 구성 완료 확인                           │
│  □ VPC Connectivity Profile에서 Auto SNAT 활성화 확인             │
│                                                                 │
│  컴퓨팅:                                                         │
│  □ ESX 호스트 이름이 소문자인지 확인                                │
│  □ vSphere DRS: Fully Automate 모드 확인                        │
│  □ vSphere HA 활성화 확인                                        │
│  □ NSX Edge: Large 폼팩터, Active/Standby 확인                  │
│                                                                 │
│  스토리지:                                                       │
│  □ 데이터스토어가 클러스터 내 모든 ESX 호스트와 공유되는지 확인       │
│  □ 3-Zone: Topology Aware 정책 생성 확인                         │
│  □ 스토리지 정책에 올바른 태그가 매핑되었는지 확인                    │
│                                                                 │
│  로드 밸런서:                                                     │
│  □ Supervisor와 VPC가 동일한 LB 유형을 사용하는지 확인              │
│  □ Avi 사용 시: Avi SE용 /26 서브넷 예약 확인                     │
│                                                                 │
│  DNS/NTP:                                                       │
│  □ vCenter가 FQDN 사용 시 DNS 서버 설정 확인                     │
│  □ NTP 서버 접근 가능 확인                                        │
│                                                                 │
│  NSX:                                                           │
│  □ NSX 버전 호환성 확인 (Broadcom Product Interoperability Matrix)│
│  □ IPv6 클러스터 불가 (vSphere 9.0 Supervisor)                   │
│  □ Centralized Gateway 구성 완료 확인                             │
│  □ Transit Gateway HA Mode = Active/Standby 확인                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 자주 발생하는 문제

| 증상 | 원인 | 해결 |
|------|------|------|
| Namespace 생성 불가 | Centralized Gateway 미구성 | Edge 노드 배포 후 Centralized Gateway 구성 |
| VPC 서브넷 할당 실패 | Private VPC CIDR 부족 | CIDR 블록 확대 또는 기존 할당 정리 |
| 외부 접근 불가 | External IP Block 충돌 | 물리 NW, VLAN PG, NSX 오버레이와 겹치지 않는 IP 사용 |
| LB 서비스 실패 | Supervisor/VPC LB 불일치 | 동일한 LB 유형으로 통일 |
| CP VM 통신 불가 | 관리 네트워크 IP 부족 | 연속 5개 IP 확보 확인 |

## 14. 정리 및 요약

### 핵심 포인트

```
┌─────────────────────────────────────────────────────────────────┐
│                      핵심 요약                                    │
│                                                                 │
│  1. VPC는 VCF 9.0에서 Supervisor의 권장 네트워크 스택              │
│     → VCF Automation 환경의 표준 워크로드 네트워크                  │
│                                                                 │
│  2. NSX Project → VPC → vSphere Namespace 계층 구조              │
│     → 각 VPC는 독립적인 라우팅 도메인                               │
│     → NCP가 System/Service/Namespace VPC를 자동 생성              │
│                                                                 │
│  3. 3가지 서브넷 유형 이해 필수                                     │
│     → Private (내부), External (외부), Private TGW (VPC간)       │
│                                                                 │
│  4. Centralized Transit Gateway = 외부 연결의 핵심                 │
│     → NSX Edge 노드 + Tier-0 GW를 통한 트래픽 라우팅              │
│     → Active/Standby HA 모드 필수                                │
│                                                                 │
│  5. 스토리지 정책은 배포 전 반드시 생성                              │
│     → 태그 기반 배치 규칙 사용                                     │
│     → 3-Zone은 Topology Aware 정책 필수                          │
│                                                                 │
│  6. LB는 Supervisor와 VPC가 반드시 동일 유형 사용                  │
│     → NSX LB (기본) 또는 Avi LB (선택)                            │
│                                                                 │
│  7. CP 크기는 Scale Up만 가능                                     │
│     → 초기 사이징이 중요!                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
