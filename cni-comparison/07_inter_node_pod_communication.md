# 07. 노드 간 파드 통신 다이어그램 비교

## 개요

Kubernetes에서 서로 다른 노드에 위치한 파드 간 통신은 CNI 플러그인이 담당합니다.
각 CNI는 고유한 방식으로 패킷을 캡슐화하고 전달합니다.
이 문서는 **Antrea(OVS/VXLAN)**, **Calico(BGP Direct / VXLAN)**, **Cilium(eBPF/VXLAN)**의
노드 간 파드 통신 흐름을 다이어그램으로 비교합니다.

---

## 1. Antrea — OVS + VXLAN 오버레이

Antrea는 Open vSwitch(OVS)를 데이터 플레인으로 사용하며,
기본적으로 **VXLAN(또는 Geneve) 오버레이** 터널로 노드 간 통신을 수행합니다.

### 패킷 흐름

```
┌──────────────────────────────────────────────────────────────────┐
│                          Node A (10.1.1.1)                       │
│                                                                  │
│  ┌─────────────┐    ┌──────────────────────────────────────┐     │
│  │   Pod A     │    │        Open vSwitch (OVS)            │     │
│  │ 10.244.1.10 │    │                                      │     │
│  │             │    │  Table 0: 분류                       │     │
│  │  eth0 ──────┼────┼► Table 10: VLAN/Port 매핑           │     │
│  │             │    │  Table 20: L2 포워딩                 │     │
│  │             │    │  Table 70: 터널 출력                 │     │
│  └─────────────┘    │        │                             │     │
│                     └────────┼─────────────────────────────┘     │
│                              │ VXLAN 캡슐화                      │
│                              │ (원본 패킷 → UDP/4789 래핑)       │
│                              ▼                                   │
│                     ┌────────────────┐                           │
│                     │ antrea-tun0    │  (VXLAN 터널 인터페이스)  │
│                     └───────┬────────┘                           │
└───────────────────────────── │ ────────────────────────────────── ┘
                               │
                 물리 네트워크 (Underlay)
                 Outer: src=10.1.1.1, dst=10.1.1.2
                 Inner: src=10.244.1.10, dst=10.244.2.20
                               │
┌───────────────────────────── │ ────────────────────────────────── ┐
│                          Node B (10.1.1.2)                       │
│                              │                                   │
│                     ┌────────▼───────┐                           │
│                     │ antrea-tun0    │  VXLAN 디캡슐화           │
│                     └───────┬────────┘                           │
│                             │                                    │
│                     ┌───────▼──────────────────────────────┐     │
│                     │        Open vSwitch (OVS)            │     │
│                     │                                      │     │
│                     │  Table 0: 터널 패킷 수신             │     │
│                     │  Table 30: 목적지 파드 포트 결정      │     │
│  ┌─────────────┐    │  Table 80: L2/L3 포워딩             │     │
│  │   Pod B     │    │        │                             │     │
│  │ 10.244.2.20 │    └────────┼─────────────────────────────┘     │
│  │             │             │                                    │
│  │  eth0 ◄─────┼─────────────┘                                   │
│  └─────────────┘                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### VXLAN 패킷 구조

```
┌─────────────────────────────────────────────────────────────────┐
│              VXLAN 캡슐화된 패킷 (Antrea)                        │
├──────────────┬──────────────┬──────────────┬────────────────────┤
│  Outer Eth   │  Outer IP    │  Outer UDP   │  VXLAN Header      │
│  (노드 MAC)  │ src:10.1.1.1 │  dport:4789  │  VNI: 터널 ID      │
│              │ dst:10.1.1.2 │              │                    │
├──────────────┴──────────────┴──────────────┴────────────────────┤
│                        Inner (원본 패킷)                         │
│  Inner Eth  │  Inner IP                  │  TCP/UDP  │  Payload │
│  (Pod MAC)  │ src:10.244.1.10            │           │          │
│             │ dst:10.244.2.20            │           │          │
└─────────────┴────────────────────────────┴───────────┴──────────┘
```

### 단계별 흐름

```
① Pod A (10.244.1.10) → 목적지 IP 10.244.2.20 패킷 생성
② veth pair → OVS 브릿지 진입
③ OVS 플로우 테이블: 목적지가 원격 노드 서브넷 → 터널 출력 포트로 포워딩
④ antrea-agent: 목적지 노드 IP(10.1.1.2) 조회 (antrea-controller에서 수신한 정보)
⑤ VXLAN 캡슐화: Inner 패킷을 UDP/4789로 래핑, Outer dst = 10.1.1.2
⑥ 물리 NIC → 물리 네트워크 전송
⑦ Node B의 antrea-tun0 수신 → VXLAN 디캡슐화
⑧ OVS 플로우 테이블: Inner dst(10.244.2.20) → Pod B 포트로 포워딩
⑨ veth pair → Pod B (10.244.2.20) 도달
```

## 2. Calico — 모드별 비교

Calico는 운영 환경에 따라 세 가지 노드 간 통신 모드를 지원합니다.

### 2-A. BGP Direct 라우팅 (오버레이 없음 — 최고 성능)

```
┌──────────────────────────────────────────────────────────────────┐
│                          Node A (10.1.1.1)                       │
│  Routing Table:                                                  │
│  10.244.2.0/26 via 10.1.1.2  ← BIRD(BGP)가 자동 주입            │
│                                                                  │
│  ┌─────────────┐    ┌──────────────┐    ┌──────────────────┐    │
│  │   Pod A     │    │  veth pair   │    │  Linux Routing   │    │
│  │ 10.244.1.10 ├────► cali1a2b3c4d ├────►  Table (Felix   │    │
│  │             │    │              │    │   가 관리)        │    │
│  └─────────────┘    └──────────────┘    └───────┬──────────┘    │
│                                                  │              │
│  ┌──────────────────────────────────────────┐    │              │
│  │  BIRD (BGP 데몬)                         │    │              │
│  │  - AS 내 iBGP / 피어 노드와 경로 교환    │    │              │
│  │  - 10.244.1.0/26 광고                   │    │              │
│  └──────────────────────────────────────────┘    │              │
└─────────────────────────────────────────── │ ─── ┘
                                             │
               물리 네트워크 (캡슐화 없음!)
               IP: src=10.244.1.10, dst=10.244.2.20
               (원본 패킷 그대로 전송)
                                             │
┌──────────────────────────────────────────── │ ───────────────────┐
│                          Node B (10.1.1.2)  │                    │
│  Routing Table:                             │                    │
│  10.244.1.0/26 via 10.1.1.1  ← BGP 수신    │                    │
│                                             ▼                    │
│                     ┌───────────────────────────────────┐        │
│                     │  Linux Routing Table               │        │
│                     │  dst=10.244.2.20 → cali5e6f7g8h  │        │
│                     └──────────────────┬────────────────┘        │
│                                        │                         │
│  ┌─────────────┐    ┌──────────────────▼──┐                      │
│  │   Pod B     │    │     veth pair        │                      │
│  │ 10.244.2.20 │◄───┤  cali5e6f7g8h        │                      │
│  └─────────────┘    └──────────────────────┘                      │
└──────────────────────────────────────────────────────────────────┘
```

**BGP Direct 패킷 구조 (캡슐화 없음):**

```
┌──────────────────────────────────────────────┐
│           원본 패킷 그대로 전송               │
├──────────────┬──────────────────┬─────────────┤
│  Ethernet    │       IP         │  TCP/UDP    │
│  (물리 MAC)  │ src: 10.244.1.10 │  + Payload  │
│              │ dst: 10.244.2.20 │             │
└──────────────┴──────────────────┴─────────────┘
※ 오버레이 헤더 없음 → 최소 오버헤드
```

## 3. Cilium — eBPF + VXLAN / Direct 라우팅

Cilium은 eBPF 프로그램을 커널에 직접 주입하여 패킷을 처리합니다.
오버레이(VXLAN)와 네이티브 라우팅 모드 모두 지원합니다.

### 3-A. VXLAN 모드 (eBPF + 오버레이)

```
┌──────────────────────────────────────────────────────────────────┐
│                          Node A (10.1.1.1)                       │
│                                                                  │
│  ┌─────────────┐                                                 │
│  │   Pod A     │   ┌────────────────────────────────────────┐   │
│  │ 10.244.1.10 │   │           eBPF 처리 경로               │   │
│  │             │   │                                        │   │
│  │  eth0 ──────┼───►  TC ingress hook (eBPF 프로그램)       │   │
│  │             │   │   ① 목적지 IP 조회 (BPF Map)           │   │
│  │             │   │   ② 원격 노드 터널 엔드포인트 결정      │   │
│  │             │   │   ③ VXLAN 캡슐화 (커널 내 처리)        │   │
│  └─────────────┘   │         │                              │   │
│                    └─────────┼──────────────────────────────┘   │
│                              │                                  │
│                    ┌─────────▼──────────┐                       │
│                    │  cilium_vxlan       │  (VXLAN 디바이스)    │
│                    │  UDP/8472          │  ← Cilium 기본 포트   │
│                    └─────────┬──────────┘                       │
└──────────────────────────────│──────────────────────────────────┘
                               │
            물리 네트워크
            Outer: src=10.1.1.1, dst=10.1.1.2, UDP/8472
            Inner: src=10.244.1.10, dst=10.244.2.20
                               │
┌──────────────────────────────│──────────────────────────────────┐
│                          Node B (10.1.1.2)                       │
│                              │                                  │
│                    ┌─────────▼──────────┐                       │
│                    │  cilium_vxlan       │  VXLAN 수신           │
│                    └─────────┬──────────┘                       │
│                              │                                  │
│                    ┌─────────▼──────────────────────────────┐   │
│                    │        eBPF 처리 경로                   │   │
│                    │   ① TC ingress hook: 디캡슐화           │   │
│                    │   ② BPF Map으로 목적지 파드 엔드포인트   │   │
│                    │      조회 (Identity 기반)               │   │
│                    │   ③ NetworkPolicy BPF 프로그램 실행     │   │
│                    │   ④ 허용 시 목적지 veth로 전달          │   │
│                    └─────────┬──────────────────────────────┘   │
│                              │                                  │
│  ┌─────────────┐    ┌─────────▼────────┐                        │
│  │   Pod B     │    │  lxc veth pair    │                        │
│  │ 10.244.2.20 │◄───┤ (lxcXXXXXXXX)    │                        │
│  └─────────────┘    └──────────────────┘                        │
└──────────────────────────────────────────────────────────────────┘
```

### eBPF BPF Map 구조

```
┌─────────────────────────────────────────────────────┐
│              Cilium BPF Map (커널 내)                │
├─────────────────────────┬───────────────────────────┤
│  cilium_ipcache         │ IP → Identity 매핑         │
│  10.244.2.20 → ID:1234  │                           │
├─────────────────────────┼───────────────────────────┤
│  cilium_tunnel_map      │ 노드 IP → 터널 엔드포인트  │

│    tunnel: 10.1.1.2     │                           │
├─────────────────────────┼───────────────────────────┤
│  cilium_policy          │ Identity 쌍별 허용/차단    │
│  ID:1111 → ID:1234 →    │                           │
│    Allow TCP:80         │                           │
└─────────────────────────┴───────────────────────────┘
```

### 3-B. 네이티브 라우팅 모드 (eBPF Direct)

kube-router 또는 BGP(bird)와 연동하여 오버레이 없이 직접 라우팅합니다.

```
┌──────────────────────────────────────────────────────────────────┐
│                          Node A (10.1.1.1)                       │
│                                                                  │
│  ┌─────────────┐    ┌─────────────────────────────────────────┐  │
│  │   Pod A     │    │         eBPF TC hook (lxc veth)         │  │
│  │ 10.244.1.10 ├────►  ① 목적지 10.244.2.20 → 원격 노드임    │  │
│  │             │    │  ② BPF fib_lookup: 라우팅 테이블 조회  │  │
│  │             │    │  ③ 다음 홉 = 10.1.1.2 결정             │  │
│  │             │    │  ④ 패킷 헤더 재작성 (MAC 변경)          │  │
│  └─────────────┘    └──────────────────┬────────────────────── ┘  │
│                                        │ 캡슐화 없음!             │
│                                        ▼                         │
│                     ┌──────────────────────────┐                 │
│                     │  물리 NIC                 │                 │
│                     │  IP: src=10.244.1.10     │                 │
│                     │      dst=10.244.2.20     │                 │
│                     └──────────────────────────┘                 │
└──────────────────────────────────────────────────────────────────┘
                               │
             물리 네트워크 (원본 IP 그대로)
                               │
┌──────────────────────────────────────────────────────────────────┐
│                          Node B (10.1.1.2)                       │
│                                                                  │
│                     ┌──────────────────────────┐                 │
│                     │  물리 NIC 수신            │                 │
│                     └──────────────┬───────────┘                 │
│                                    │                             │
│                    ┌───────────────▼───────────────────────────┐ │
│                    │  eBPF XDP or TC hook                      │ │
│                    │  ① ipcache에서 Identity 조회              │ │
│                    │  ② NetworkPolicy 검사 (BPF 프로그램)      │ │
│                    │  ③ lxc veth 포워딩                        │ │
│                    └───────────────┬───────────────────────────┘ │
│                                    │                             │
│  ┌─────────────┐    ┌──────────────▼──────┐                      │
│  │   Pod B     │◄───┤   lxc veth pair     │                      │
│  │ 10.244.2.20 │    └─────────────────────┘                      │
└──────────────────────────────────────────────────────────────────┘
```

### 3-C. 같은 노드 파드 간 최적화 (Socket-level Bypass)

```
┌──────────────────────────────────────────────────────────────────┐
│                   Node A (같은 노드 내 통신)                      │
│                                                                  │
│  ┌─────────────┐         eBPF sockops hook          ┌──────────┐ │
│  │   Pod A     │  ──────────────────────────────►   │  Pod B   │ │
│  │ 10.244.1.10 │  네트워크 스택 전체 우회!           │10.244.1.2│ │
│  └─────────────┘  (소켓 레벨에서 직접 연결)          └──────────┘ │
│                                                                  │
│  ※ 같은 노드 파드 간 통신은 veth/OVS/터널을 거치지 않음         │
│    → 지연 시간 최소화, 처리량 극대화                             │
└──────────────────────────────────────────────────────────────────┘
```

## 4. 전체 비교 다이어그램

```
┌──────────────────────────────────────────────────────────────────┐
│              노드 간 파드 통신 방식 한눈에 보기                   │
│                                                                  │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━     │
│  Antrea (OVS + VXLAN)                                            │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━     │
│                                                                  │
│  Pod A → [veth] → [OVS] → [VXLAN 캡슐화] → 물리NIC             │
│                                        → [물리NIC] → [VXLAN 해제] → [OVS] → [veth] → Pod B │
│                                                                  │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━     │
│  Calico BGP (Direct 라우팅)                                      │
```

## 6. 종합 비교표

| 항목 | Antrea (VXLAN) | Calico BGP | Calico VXLAN | Cilium VXLAN | Cilium Native |
|------|---------------|------------|--------------|--------------|---------------|
| 데이터 플레인 | OVS | Linux Routing | Linux Routing | eBPF | eBPF |
| 캡슐화 방식 | VXLAN | 없음 | VXLAN | VXLAN | 없음 |
| 캡슐화 포트 | UDP/4789 | - | UDP/4789 | UDP/8472 | - |
| 오버헤드 | 50B | 0B | 50B | 50B | 0B |
| BGP 필요 여부 | ❌ | ✅ BIRD | ❌ | 선택 | 선택 |
| 패킷 처리 위치 | OVS 커널 모듈 | Netfilter | Netfilter | eBPF JIT | eBPF JIT |
| NetworkPolicy 위치 | OVS 플로우 | iptables/eBPF | iptables/eBPF | eBPF Map | eBPF Map |
| 동일 노드 최적화 | OVS 내부 | veth 단락 | veth 단락 | Socket Bypass | Socket Bypass |
| MTU 고려 | 1450 (VXLAN) | 1500 | 1450 (VXLAN) | 1450 (VXLAN) | 1500 |
| 확장성 | 중간 | 높음 | 중간 | 매우 높음 | 매우 높음 |

## 7. 디버깅 명령어

### Antrea 디버깅

```bash
# OVS 플로우 테이블 확인
kubectl exec -n kube-system antrea-agent-XXXXX -- ovs-ofctl dump-flows br-int

# 특정 파드의 OVS 포트 확인
kubectl exec -n kube-system antrea-agent-XXXXX -- ovs-vsctl show

# antctl로 플로우 추적
antctl trace-packet --source-pod ns/pod-a --destination-pod ns/pod-b
```

### Calico BGP 디버깅

```bash
# BGP 피어 상태 확인
calicoctl node status

# BGP 경로 테이블 확인
kubectl exec -n calico-system calico-node-XXXXX -- bird -s /var/run/calico/bird.ctl "show route"

# Felix 로그 확인
kubectl logs -n calico-system calico-node-XXXXX -c calico-node | grep felix

# 라우팅 테이블 확인 (노드에서)
ip route show | grep cali
```

### Cilium eBPF 디버깅

```bash
# 파드 간 연결 테스트
cilium connectivity test

# BPF Map 확인 (ipcache)
kubectl exec -n kube-system cilium-XXXXX -- cilium map get cilium_ipcache

# 터널 엔드포인트 확인
kubectl exec -n kube-system cilium-XXXXX -- cilium map get cilium_tunnel_map

# 실시간 패킷 흐름 추적 (Hubble)
hubble observe --from-pod ns/pod-a --to-pod ns/pod-b --follow

# eBPF 프로그램 목록 확인
kubectl exec -n kube-system cilium-XXXXX -- bpftool prog list
```
