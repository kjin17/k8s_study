iptables 모드 (kube-proxy)

개요

- kube-proxy가 iptables 모드로 동작하면 Kubernetes 서비스의 가상 IP(ClusterIP)를 위해 노드의 netfilter(iptables) 규칙을 생성한다.
- 각 서비스/엔드포인트는 NAT(주로 DNAT)를 사용하는 규칙 체인으로 표현되며, 패킷이 도착하면 규칙을 순차 검사하여 대상 엔드포인트로 전달한다.

장점

- 높은 호환성: 대부분의 리눅스 배포판에서 기본 제공되는 netfilter/iptables 사용.
- 설정·디버깅이 상대적으로 쉬움: iptables-save/ip6tables-save, iptables -L 등으로 규칙 확인 가능.
- 단순성: 개념적으로 이해하기 쉬워 운영자가 문제 파악하기 용이.
- 연결 추적(conntrack)을 활용해 NAT 기반 세션 유지 가능.

단점

- 확장성 한계: 서비스 수와 엔드포인트 수가 많아지면 규칙 수가 급증 → 룰 체인 길어짐 → 커널에서 룰 순차 검사 비용 상승.
- 성능: 대량의 동시 연결/고빈도 트래픽에서 CPU 오버헤드 증가 및 지연이 발생할 수 있음.
- 규칙 동기화 비용: kube-proxy가 규칙을 갱신할 때 사용자공간↔커널 작업 비용이 크며, 잦은 엔드포인트 변경 시 부담.

운영 팁

- 소규모~중간 규모 클러스터에는 여전히 안정적이고 무난한 선택.
- 노드에서 ipset을 함께 사용해 대규모 IP 집합을 묶어 관리하면 규칙 수를 줄일 수 있음.
- 성능 문제 탐지 시 ipvs 모드 전환을 고려.
