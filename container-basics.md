## 1. 컨테이너란 무엇인가?

### 한 문장 정의

> **컨테이너**는 애플리케이션과 그 실행에 필요한 모든 것(코드, 라이브러리, 설정 파일)을 하나로 묶어서, 어떤 환경에서든 동일하게 실행할 수 있도록 격리된 공간을 제공하는 기술입니다.

### 비유로 이해하기

```
┌─────────────────────────────────────────────────────────┐
│                    화물 운송의 혁명                        │
│                                                         │
│   [컨테이너 등장 전]              [컨테이너 등장 후]        │
│                                                         │
│   🍎 사과 따로                  ┌──────────────┐         │
│   📺 TV 따로          →        │ 🍎📺🪑 표준 박스 │         │
│   🪑 의자 따로                  └──────────────┘         │
│                                                         │
│   화물마다 다른 취급 필요           어떤 트럭, 배, 기차든    │
│   깨지고, 섞이고, 느림             동일한 방식으로 운송      │
└─────────────────────────────────────────────────────────┘
```

해운업에서 **표준화된 컨테이너 박스**가 물류를 혁신한 것처럼, IT에서도 **소프트웨어 컨테이너**가 애플리케이션 배포를 혁신했습니다.

| 해운 컨테이너 | 소프트웨어 컨테이너 |
|---------------|---------------------|
| 화물을 표준 박스에 넣음 | 앱 + 라이브러리를 이미지로 패키징 |
| 어떤 선박/트럭에든 실을 수 있음 | 어떤 서버/클라우드에든 실행 가능 |
| 내용물끼리 섞이지 않음 | 프로세스끼리 격리됨 |
| 규격이 같아서 쌓을 수 있음 | 동일한 방식으로 대량 배포 가능 |

## 2. 왜 컨테이너가 필요한가?

### "내 컴퓨터에선 되는데?" 문제

개발자가 가장 많이 하는 말 중 하나입니다.

```
개발자 노트북                    운영 서버
┌──────────────┐               ┌──────────────┐
│ Python 3.11  │               │ Python 3.8   │  ← 버전 다름
│ Ubuntu 22.04 │               │ CentOS 7     │  ← OS 다름
│ libssl 3.0   │               │ libssl 1.1   │  ← 라이브러리 다름
│              │               │              │
│  ✅ 앱 정상   │               │  ❌ 앱 오류   │
└──────────────┘               └──────────────┘
```

**컨테이너가 이 문제를 해결합니다:**

```
어디서든 동일한 컨테이너 실행
┌──────────────────────────┐
│  컨테이너                  │
│  ┌────────────────────┐  │
│  │ Python 3.11        │  │
│  │ libssl 3.0         │  │
│  │ 앱 코드 + 설정      │  │
│  │                    │  │
│  │ ✅ 항상 동일하게 동작 │  │
│  └────────────────────┘  │
└──────────────────────────┘
  개발 노트북 ✅  테스트 서버 ✅  운영 서버 ✅  클라우드 ✅
```

### 컨테이너가 해결하는 핵심 문제들

| 문제 | 기존 방식 | 컨테이너 방식 |
|------|-----------|---------------|
| 환경 불일치 | 서버마다 수동으로 세팅 | 이미지 하나로 어디서든 동일 실행 |
| 의존성 충돌 | A앱은 Java 8, B앱은 Java 17 필요 | 각 컨테이너가 자기만의 Java 보유 |
| 배포 속도 | 서버 세팅에 수 시간 | 컨테이너 시작에 수 초 |
| 자원 낭비 | 앱마다 VM 한 대씩 | 한 서버에 수십 개 컨테이너 실행 |
| 확장성 | 서버 추가 구매 → 설치 → 설정 | 컨테이너 복제만 하면 끝 |

## 3. 가상머신 vs 컨테이너

### 구조 비교

```
         가상머신 (VM)                         컨테이너

┌──────┐ ┌──────┐ ┌──────┐       ┌──────┐ ┌──────┐ ┌──────┐
│ App1 │ │ App2 │ │ App3 │       │ App1 │ │ App2 │ │ App3 │
├──────┤ ├──────┤ ├──────┤       ├──────┤ ├──────┤ ├──────┤
│ Bins │ │ Bins │ │ Bins │       │ Bins │ │ Bins │ │ Bins │
│ Libs │ │ Libs │ │ Libs │       │ Libs │ │ Libs │ │ Libs │
├──────┤ ├──────┤ ├──────┤       └──┬───┘ └──┬───┘ └──┬───┘
│Guest │ │Guest │ │Guest │          │        │        │
│  OS  │ │  OS  │ │  OS  │       ┌──┴────────┴────────┴───┐
├──────┤ ├──────┤ ├──────┤       │    컨테이너 런타임       │
└──┬───┘ └──┬───┘ └──┬───┘       ├─────────────────────────┤
┌──┴────────┴────────┴───┐       │      호스트 OS (리눅스)   │
│      하이퍼바이저        │       ├─────────────────────────┤
├─────────────────────────┤       │        하드웨어           │
│      호스트 OS           │       └─────────────────────────┘
├─────────────────────────┤
│        하드웨어           │
└─────────────────────────┘

 각 VM마다 완전한 OS 포함           호스트 OS의 커널을 공유
 무겁고 느림 (GB 단위)              가볍고 빠름 (MB 단위)
```

### 상세 비교

| 항목 | 가상머신 (VM) | 컨테이너 |
|------|---------------|----------|
| **격리 수준** | 완전한 격리 (별도 OS) | 프로세스 수준 격리 (커널 공유) |
| **시작 시간** | 수 분 | 수 초 |
| **이미지 크기** | 수 GB ~ 수십 GB | 수 MB ~ 수백 MB |
| **성능 오버헤드** | 높음 (하이퍼바이저 경유) | 거의 없음 (네이티브에 가까움) |
| **실행 밀도** | 서버당 수 ~ 수십 개 | 서버당 수백 ~ 수천 개 |
| **OS** | 각 VM마다 독립 OS | 호스트 커널 공유 |
| **사용 사례** | OS가 달라야 할 때, 강한 격리 필요 시 | 마이크로서비스, CI/CD, 빠른 배포 |

### 언제 무엇을 쓰는가?

```
"리눅스 앱을 빠르고 가볍게 배포하고 싶다"     → 컨테이너 ✅
"Windows와 Linux를 한 서버에서 돌려야 한다"   → VM ✅
"보안상 완전한 격리가 필수다"                  → VM ✅ (또는 VM + 컨테이너 조합)
"수백 개의 마이크로서비스를 운영한다"           → 컨테이너 ✅
```

> 실무에서는 VM과 컨테이너를 함께 사용하는 경우가 많습니다.
> 예: VM 위에 리눅스를 설치하고, 그 위에서 컨테이너를 실행

## 4. 컨테이너를 가능하게 하는 리눅스 기술

컨테이너는 마법이 아니라, **리눅스 커널이 제공하는 기능들의 조합**입니다.

### 핵심 기술 3가지

```
┌─────────────────────────────────────────────────────┐
│                    리눅스 컨테이너                     │
│                                                     │
│   ┌─────────────┐  ┌──────────┐  ┌──────────────┐  │
│   │ Namespace   │  │ cgroup   │  │ Union FS     │  │
│   │ (격리)      │  │ (자원제한) │  │ (파일시스템)  │  │
│   └─────────────┘  └──────────┘  └──────────────┘  │
│                                                     │
│   "뭘 볼 수 있나"   "얼마나 쓸 수   "파일을 어떻게   │
│                      있나"          구성하나"        │
└─────────────────────────────────────────────────────┘
```

### 4-1. Namespace (네임스페이스) - "격리"

네임스페이스는 프로세스가 **볼 수 있는 범위를 제한**합니다.

```
┌─── 호스트 시스템 ────────────────────────────────┐
│                                                 │
│  ┌─── 컨테이너 A ──────┐  ┌─── 컨테이너 B ────┐  │
│  │ PID 1: nginx        │  │ PID 1: python    │  │
│  │ PID 2: worker       │  │ PID 2: celery    │  │
│  │                     │  │                  │  │
│  │ 네트워크: 172.17.0.2 │  │ 네트워크: 172.17.0.3│ │
│  │ 호스트명: web-server │  │ 호스트명: api-server│ │
│  │                     │  │                  │  │
│  │ A는 B를 볼 수 없음   │  │ B는 A를 볼 수 없음 │  │
│  └─────────────────────┘  └──────────────────┘  │
│                                                 │
│  호스트에서는 모든 프로세스가 보임                    │
│  PID 1234: nginx (컨테이너 A)                     │
│  PID 1235: worker (컨테이너 A)                    │
│  PID 5678: python (컨테이너 B)                    │
│  PID 5679: celery (컨테이너 B)                    │
└─────────────────────────────────────────────────┘
```

리눅스가 제공하는 네임스페이스 종류:

| 네임스페이스 | 격리 대상 | 설명 |
|-------------|-----------|------|
| **PID** | 프로세스 ID | 컨테이너 안에서 PID 1부터 시작 |
| **Network** | 네트워크 | 독립된 IP, 포트, 라우팅 테이블 |
| **Mount** | 파일시스템 | 독립된 마운트 포인트 |
| **UTS** | 호스트명 | 컨테이너별 독립 호스트명 |
| **IPC** | 프로세스 간 통신 | 공유 메모리, 세마포어 격리 |
| **User** | 사용자/그룹 ID | 컨테이너 내 root ≠ 호스트 root |
| **Cgroup** | cgroup 루트 | 자원 제한 계층 격리 |

### 4-2. cgroup (Control Groups) - "자원 제한"

cgroup은 컨테이너가 **사용할 수 있는 자원의 양을 제한**합니다.

```
물리 서버 자원: CPU 8코어, 메모리 32GB
┌─────────────────────────────────────────┐
│                                         │
│  ┌─ 컨테이너 A ──┐  ┌─ 컨테이너 B ──┐   │
│  │ CPU: 2코어    │  │ CPU: 4코어    │   │
│  │ 메모리: 4GB   │  │ 메모리: 8GB   │   │
│  │ 디스크I/O:    │  │ 디스크I/O:    │   │
│  │  100MB/s     │  │  200MB/s     │   │
│  └──────────────┘  └──────────────┘   │
│                                         │
│  → 컨테이너 A가 아무리 바빠도             │
│    CPU 2코어, 메모리 4GB를 넘길 수 없음   │
│  → 한 컨테이너가 서버 자원을 독점하는 것   │
│    을 방지                               │
└─────────────────────────────────────────┘
```

cgroup으로 제한할 수 있는 자원:

| 자원 | 설명 | 예시 |
|------|------|------|
| **CPU** | 사용 가능한 CPU 시간 | 최대 2코어 |
| **Memory** | 사용 가능한 메모리 양 | 최대 512MB |
| **Block I/O** | 디스크 읽기/쓰기 속도 | 최대 100MB/s |
| **Network** | 네트워크 대역폭 | 최대 1Gbps |
| **PIDs** | 생성 가능한 프로세스 수 | 최대 100개 |

### 4-3. Union File System - "이미지 레이어"

컨테이너 이미지는 **여러 레이어를 겹쳐서** 하나의 파일시스템처럼 보이게 합니다.
(다음 장에서 더 자세히 다룹니다.)

## 5. 컨테이너 이미지

### 이미지란?

> **컨테이너 이미지**는 컨테이너를 실행하기 위한 **읽기 전용 템플릿**입니다.
> 애플리케이션 코드, 런타임, 라이브러리, 환경 변수, 설정 파일이 모두 포함됩니다.

```
비유:

  이미지 (Image)          컨테이너 (Container)
  ─────────────          ────────────────────
  붕어빵 틀               붕어빵
  설계도                  건물
  클래스 (Class)          인스턴스 (Instance)

  → 하나의 이미지로 여러 컨테이너를 실행할 수 있음
```

### 이미지 레이어 구조

이미지는 여러 개의 **읽기 전용 레이어**로 구성됩니다.

```
┌─────────────────────────────────────┐
│        컨테이너 (실행 중)             │
│  ┌─────────────────────────────┐    │
│  │  읽기/쓰기 레이어 (R/W)      │ ← 컨테이너에서 변경한 내용
│  │  (컨테이너 삭제 시 사라짐)    │    이 레이어에 기록됨
│  ├─────────────────────────────┤    │
│  │  Layer 4: COPY app.py       │ ← 앱 코드 복사
│  ├─────────────────────────────┤    │
│  │  Layer 3: RUN pip install   │ ← 패키지 설치
│  ├─────────────────────────────┤    │
│  │  Layer 2: RUN apt-get update│ ← OS 패키지 업데이트
│  ├─────────────────────────────┤    │
│  │  Layer 1: Ubuntu 22.04 Base │ ← 베이스 이미지
│  └─────────────────────────────┘    │
│                                     │
│  모든 레이어가 합쳐져서 하나의         │
│  파일시스템처럼 보임                   │
└─────────────────────────────────────┘
```

### 레이어의 장점

```
이미지 A (Node.js 앱)           이미지 B (Python 앱)
┌───────────────────┐          ┌───────────────────┐
│ Layer: app.js     │          │ Layer: app.py     │
├───────────────────┤          ├───────────────────┤
│ Layer: npm install│          │ Layer: pip install│
├───────────────────┤          ├───────────────────┤
│ Layer: Ubuntu     │ ◄─공유──► │ Layer: Ubuntu     │
└───────────────────┘          └───────────────────┘

→ Ubuntu 레이어는 디스크에 한 번만 저장됨
→ 두 이미지가 같은 베이스 레이어를 공유하여 저장 공간 절약
```

### 컨테이너 레지스트리

이미지를 저장하고 공유하는 저장소입니다.

| 레지스트리 | 설명 | URL |
|-----------|------|-----|
| **Docker Hub** | 가장 큰 퍼블릭 레지스트리 | hub.docker.com |
| **GitHub Container Registry** | GitHub 연동 | ghcr.io |
| **Amazon ECR** | AWS 프라이빗 레지스트리 | |
| **Google Artifact Registry** | GCP 프라이빗 레지스트리 | |
| **Harbor** | 오픈소스 프라이빗 레지스트리 | |

### 이미지 이름 규칙

```
레지스트리/이름:태그

예시:
  nginx                        → docker.io/library/nginx:latest (공식 이미지)
  nginx:1.25                   → docker.io/library/nginx:1.25
  mycompany/myapp:v2.1.0       → docker.io/mycompany/myapp:v2.1.0
  ghcr.io/org/app:main         → GitHub Container Registry의 이미지

주의: "latest" 태그는 "최신"이 아니라 "태그 미지정 시 기본값"입니다.
      운영 환경에서는 반드시 구체적인 버전 태그를 사용하세요.
```

## 6. 컨테이너 런타임

### 컨테이너 런타임이란?

> 컨테이너를 실제로 **생성하고 실행하는 소프트웨어**입니다.

```
사용자 명령 → 컨테이너 런타임 → 리눅스 커널 기능 호출
             (containerd 등)    (namespace, cgroup 등)
```

### 주요 컨테이너 런타임

```
┌─────────────────────────────────────────────────────┐
│                  사용자 인터페이스                      │
│          Docker CLI / nerdctl / crictl               │
├─────────────────────────────────────────────────────┤
│                고수준 런타임 (High-level)               │
│          containerd    /    CRI-O                    │
│          (Docker 내부)      (K8s 전용)                │
├─────────────────────────────────────────────────────┤
│                저수준 런타임 (Low-level)                │
│                      runc                            │
│          (실제 namespace/cgroup 생성)                  │
├─────────────────────────────────────────────────────┤
│                    리눅스 커널                         │
│           namespace, cgroup, seccomp 등              │
└─────────────────────────────────────────────────────┘
```

| 런타임 | 역할 | 설명 |
|--------|------|------|
| **runc** | 저수준 런타임 | OCI 표준 준수, 실제 컨테이너 생성 |
| **containerd** | 고수준 런타임 | 이미지 관리, 컨테이너 수명주기 관리 |
| **CRI-O** | 고수준 런타임 | Kubernetes 전용으로 설계 |
| **Docker Engine** | 통합 플랫폼 | containerd + 빌드 + 네트워크 + CLI |

### Docker의 위치

```
Docker는 "컨테이너의 대명사"이지만,
실제로는 여러 컴포넌트의 조합입니다:

  docker CLI
      │
      ▼
  Docker Daemon (dockerd)
      │
      ▼
  containerd ← 고수준 런타임
      │
      ▼
  runc ← 저수준 런타임
      │
      ▼
  리눅스 커널

Kubernetes 1.24부터 Docker 대신 containerd를 직접 사용합니다.
(Docker가 사라진 게 아니라, 중간 단계를 건너뛰게 된 것)
```

```bash
# 포트 연결 (-p 호스트포트:컨테이너포트)
docker run -d -p 8080:80 --name my-web nginx
# → 브라우저에서 http://localhost:8080 접속 가능

# 환경 변수 전달 (-e)

```

```bash
# 실행 중인 컨테이너 목록
docker ps

docker ps -a

# 컨테이너 중지
docker stop my-web

# 컨테이너 시작 (중지된 컨테이너 재시작)
docker start my-web

# 컨테이너 삭제 (중지 상태여야 함)
docker rm my-web

# 강제 삭제 (실행 중이어도)
docker rm -f my-web

# 실행 중인 컨테이너에 명령 실행
docker exec -it my-web /bin/bash

# 컨테이너 로그 확인
docker logs my-web
docker logs -f my-web   # 실시간 로그 (tail -f 와 비슷)
```

### 실습 예제: Nginx 웹 서버

```bash
# 5. 다시 접속 확인
curl http://localhost:8080
# → Hello Container!

# 6. 정리
docker rm -f webserver
```

```bash
  -e MYSQL_DATABASE=testdb \

SHOW DATABASES;

EXIT;

# 4. 정리
docker rm -f mydb
```

## 8. Dockerfile 작성법

### Dockerfile이란?

> 컨테이너 이미지를 만들기 위한 **레시피(설명서)** 파일입니다.

### 기본 구조

```dockerfile
# 1. 베이스 이미지 지정
FROM python:3.11-slim

# 2. 작업 디렉토리 설정
WORKDIR /app

# 3. 의존성 파일 복사 및 설치
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 4. 앱 코드 복사
COPY . .

# 5. 포트 노출 (문서화 목적)
EXPOSE 8000

# 6. 컨테이너 시작 시 실행할 명령
CMD ["python", "app.py"]
```

### 주요 명령어

| 명령어 | 설명 | 예시 |
|--------|------|------|
| `FROM` | 베이스 이미지 지정 | `FROM ubuntu:22.04` |
| `RUN` | 이미지 빌드 시 명령 실행 | `RUN apt-get install -y curl` |
| `COPY` | 호스트 → 이미지로 파일 복사 | `COPY app.py /app/` |
| `ADD` | COPY + 압축 해제/URL 지원 | `ADD archive.tar.gz /app/` |
| `WORKDIR` | 작업 디렉토리 설정 | `WORKDIR /app` |
| `ENV` | 환경 변수 설정 | `ENV NODE_ENV=production` |
| `EXPOSE` | 사용 포트 문서화 | `EXPOSE 3000` |
| `CMD` | 컨테이너 시작 시 기본 명령 | `CMD ["node", "server.js"]` |
| `ENTRYPOINT` | 컨테이너 시작 시 고정 명령 | `ENTRYPOINT ["java", "-jar"]` |

### CMD vs ENTRYPOINT

```
CMD: 기본 명령. docker run 시 덮어쓸 수 있음
  CMD ["python", "app.py"]
  docker run myimage              → python app.py 실행
  docker run myimage /bin/bash    → /bin/bash 실행 (CMD 무시)

ENTRYPOINT: 고정 명령. docker run 시 인자만 추가됨
  ENTRYPOINT ["python"]
  CMD ["app.py"]
  docker run myimage              → python app.py 실행
  docker run myimage test.py      → python test.py 실행
```

### 이미지 빌드 및 실행

```bash
# 이미지 빌드
docker build -t myapp:v1 .
#            -t 이름:태그  . (현재 디렉토리의 Dockerfile 사용)

# 빌드한 이미지 실행
docker run -d -p 8000:8000 myapp:v1
```

### Dockerfile 작성 모범 사례

```dockerfile
# ✅ 좋은 예시

# 가벼운 베이스 이미지 사용 (slim, alpine)
FROM python:3.11-slim

WORKDIR /app

# 자주 바뀌지 않는 것을 먼저 복사 (캐시 활용)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 자주 바뀌는 코드는 나중에 복사
COPY . .

# root가 아닌 일반 유저로 실행
RUN useradd -r appuser
USER appuser

EXPOSE 8000
CMD ["python", "app.py"]
```

```dockerfile
# ❌ 나쁜 예시

# 불필요하게 큰 베이스 이미지
FROM ubuntu:22.04

# 모든 것을 한 번에 복사 (코드 한 줄만 바꿔도 전체 재빌드)
COPY . /app

# RUN을 여러 줄로 나누면 레이어가 불필요하게 많아짐
RUN apt-get update
RUN apt-get install -y python3
RUN apt-get install -y python3-pip
# → 하나의 RUN으로 합치는 것이 좋음

# root로 실행 (보안 위험)
CMD ["python3", "/app/app.py"]
```

## 9. 컨테이너 네트워크

### 기본 개념

컨테이너는 기본적으로 **격리된 네트워크 환경**을 가집니다. Docker는 여러 네트워크 모드를 제공합니다.

### 네트워크 모드

```
1. Bridge (기본)
┌─── 호스트 ──────────────────────────────┐
│                                         │
│   ┌─ docker0 (브릿지) ─────────────┐    │
│   │  172.17.0.1                    │    │
│   │                                │    │
│   │  ┌──────────┐  ┌──────────┐   │    │
│   │  │컨테이너 A │  │컨테이너 B │   │    │
│   │  │172.17.0.2│  │172.17.0.3│   │    │
│   │  └──────────┘  └──────────┘   │    │
│   │                                │    │
│   │  A와 B는 서로 통신 가능          │    │
│   └────────────────────────────────┘    │
│                                         │
│   외부 접근: -p 옵션으로 포트 매핑 필요    │
└─────────────────────────────────────────┘

2. Host
┌─── 호스트 ──────────────────────────────┐
│   IP: 192.168.1.10                      │
│                                         │
│   ┌──────────┐                          │
│   │컨테이너   │ ← 호스트 네트워크 직접 사용 │
│   │(포트 80)  │    격리 없음               │
│   └──────────┘    포트 매핑 불필요          │
│                                         │
│   외부에서 192.168.1.10:80으로 직접 접근   │
└─────────────────────────────────────────┘

3. None
┌──────────┐
│컨테이너   │ ← 네트워크 없음 (완전 격리)
│          │    보안이 중요한 작업에 사용
└──────────┘
```

### 컨테이너 간 통신

```bash
# 사용자 정의 네트워크 생성
docker network create my-network

# 같은 네트워크에 컨테이너 연결
docker run -d --name web --network my-network nginx
docker run -d --name db  --network my-network mysql:8 -e MYSQL_ROOT_PASSWORD=pass

# → web 컨테이너에서 "db"라는 이름으로 접근 가능
docker exec web curl http://db:3306
# 사용자 정의 네트워크에서는 컨테이너 이름이 DNS로 자동 등록됨
```

### 포트 매핑

```
-p 호스트포트:컨테이너포트

┌─── 호스트 ─────────────────────────┐
│                                    │
│   외부 요청                         │
│   http://서버IP:8080               │
│        │                           │
│        ▼                           │
│   포트 8080 ──매핑──► 포트 80       │
│                     ┌──────────┐   │
│                     │ nginx    │   │
│                     │ (포트 80) │   │
│                     └──────────┘   │
└────────────────────────────────────┘

docker run -p 8080:80 nginx
docker run -p 3000:3000 node-app
docker run -p 5432:5432 postgres
```

## 10. 컨테이너 스토리지

### 컨테이너의 데이터 문제

```
⚠️ 컨테이너를 삭제하면 내부 데이터도 모두 사라집니다!

   docker run mysql → 데이터 저장 → docker rm mysql → 💥 데이터 소멸

   → 영속적인 데이터는 외부 스토리지에 저장해야 합니다.
```

### 스토리지 유형

```
1. Volume (권장)
┌─── 호스트 ──────────────────────────────┐
│                                         │
│   /var/lib/docker/volumes/mydata/       │
│        │                                │
│        ▼ (마운트)                        │
│   ┌──────────┐                          │
│   │컨테이너   │                          │
│   │ /data    │ ← 컨테이너가 삭제되어도    │
│   └──────────┘   Volume의 데이터는 유지   │
└─────────────────────────────────────────┘

2. Bind Mount
┌─── 호스트 ──────────────────────────────┐
│                                         │
│   /home/user/myproject/                 │
│        │                                │
│        ▼ (마운트)                        │
│   ┌──────────┐                          │
│   │컨테이너   │                          │
│   │ /app     │ ← 호스트의 특정 경로를     │
│   └──────────┘   직접 마운트              │
└─────────────────────────────────────────┘
```

### 실습: Volume 사용

```bash
# Volume 생성
docker volume create mydata

# Volume을 마운트하여 컨테이너 실행
docker run -d \
  --name db1 \
  -v mydata:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=pass \
  mysql:8

# 데이터 저장 후 컨테이너 삭제
docker rm -f db1

# 같은 Volume으로 새 컨테이너 실행 → 데이터 유지!
docker run -d \
  --name db2 \
  -v mydata:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=pass \
  mysql:8
```

### 실습: Bind Mount (개발용)

```bash
# 현재 디렉토리의 소스코드를 컨테이너에 마운트
docker run -d \
  -p 8080:80 \
  -v $(pwd)/html:/usr/share/nginx/html \
  nginx

# → html/index.html을 수정하면 컨테이너에 즉시 반영
# → 개발 중 코드 변경 사항을 바로 확인할 때 유용
```

## 12. 컨테이너 오케스트레이션과 Kubernetes로의 연결

### 컨테이너 하나로는 부족하다

실제 서비스는 수십~수백 개의 컨테이너가 필요합니다.

```
실제 서비스 구성 예시:

  ┌──────┐ ┌──────┐ ┌──────┐   웹 프론트엔드 (3개)
  │ Web  │ │ Web  │ │ Web  │
  └──┬───┘ └──┬───┘ └──┬───┘
     │        │        │
  ┌──┴────────┴────────┴───┐
  │      로드 밸런서         │
  └──┬────────┬────────┬───┘
     │        │        │
  ┌──┴──┐ ┌──┴──┐ ┌──┴──┐   API 서버 (3개)
  │ API │ │ API │ │ API │
  └──┬──┘ └──┬──┘ └──┬──┘
     │        │        │
  ┌──┴──┐ ┌──┴──┐ ┌──┴──┐   데이터베이스, 캐시, 큐
  │ DB  │ │Cache│ │Queue│
  └─────┘ └─────┘ └─────┘
```

### 수동 관리의 한계

```
컨테이너 10개면 수동으로 관리 가능...
컨테이너 100개가 되면?

  "컨테이너가 죽으면 누가 다시 띄우지?"
  "트래픽이 몰리면 누가 컨테이너를 늘리지?"
  "새 버전을 배포할 때 하나씩 교체해야 하나?"
  "컨테이너를 어느 서버에 배치할지 누가 결정하지?"
  "설정 파일과 비밀번호는 어떻게 관리하지?"

  → 이 모든 것을 자동으로 해주는 것이 컨테이너 오케스트레이션
```

### Kubernetes가 해결하는 문제

| 문제 | Kubernetes의 해결 방법 |
|------|----------------------|
| 컨테이너가 죽었다 | 자동으로 재시작 (Self-healing) |
| 트래픽 급증 | 자동으로 컨테이너 수 증가 (Auto-scaling) |
| 새 버전 배포 | 하나씩 교체하며 배포 (Rolling Update) |
| 서버 배치 결정 | 자원 상황에 따라 자동 배치 (Scheduling) |
| 설정/비밀 관리 | ConfigMap, Secret 오브젝트 제공 |
| 서비스 간 통신 | Service, Ingress로 네트워킹 자동 관리 |
| 스토리지 관리 | PersistentVolume으로 영속 스토리지 제공 |

## 부록: 자주 쓰는 Docker 명령어 요약

```
이미지 관련
  docker pull <이미지>        이미지 다운로드
  docker images              이미지 목록
  docker rmi <이미지>         이미지 삭제
  docker build -t <이름> .   이미지 빌드

컨테이너 관련
  docker run <이미지>         컨테이너 실행
  docker ps                  실행 중 컨테이너 목록
  docker ps -a               전체 컨테이너 목록
  docker stop <컨테이너>      컨테이너 중지
  docker start <컨테이너>     컨테이너 시작
  docker rm <컨테이너>        컨테이너 삭제
  docker logs <컨테이너>      로그 확인
  docker exec -it <컨테이너> /bin/bash    쉘 접속

볼륨 관련
  docker volume create <이름>    볼륨 생성
  docker volume ls               볼륨 목록
  docker volume rm <이름>        볼륨 삭제

네트워크 관련
  docker network create <이름>   네트워크 생성
  docker network ls              네트워크 목록

시스템 관련
  docker system df               디스크 사용량
  docker system prune            미사용 리소스 정리
```

## 부록: 용어 사전

| 용어 | 설명 |
|------|------|
| **이미지 (Image)** | 컨테이너 실행을 위한 읽기 전용 템플릿 |
| **컨테이너 (Container)** | 이미지를 실행한 인스턴스 (프로세스) |
| **레이어 (Layer)** | 이미지를 구성하는 읽기 전용 파일시스템 층 |
| **레지스트리 (Registry)** | 이미지를 저장하고 배포하는 저장소 |
| **Dockerfile** | 이미지를 만들기 위한 빌드 명세 파일 |
| **볼륨 (Volume)** | 컨테이너 외부의 영속적 데이터 저장소 |
| **바인드 마운트 (Bind Mount)** | 호스트의 특정 경로를 컨테이너에 연결 |
| **포트 매핑 (Port Mapping)** | 호스트 포트와 컨테이너 포트를 연결 |
| **네임스페이스 (Namespace)** | 프로세스 격리를 위한 리눅스 커널 기능 |
| **cgroup** | 자원 제한을 위한 리눅스 커널 기능 |
| **OCI** | Open Container Initiative, 컨테이너 표준 규격 |
| **오케스트레이션** | 다수의 컨테이너를 자동으로 관리하는 것 |
