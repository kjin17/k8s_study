# 01. 백업/복구 전략 개요

## 왜 백업이 필요한가?

Kubernetes는 빠르게 재생성되는 환경이지만, 다음과 같은 데이터는 사라지면 안 됩니다.

- 애플리케이션 설정
- 배포 리소스
- 네임스페이스 구성
- PVC/PV 데이터
- Secret / ConfigMap / CRD 상태

---

## 백업 대상 분류

### 1) Kubernetes 리소스
- Namespace
- Deployment / StatefulSet / DaemonSet
- Service / Ingress
- ConfigMap / Secret
- PVC / PV 메타데이터
- CRD / Custom Resource

### 2) 데이터 볼륨
- MySQL / PostgreSQL 데이터
- 애플리케이션 업로드 파일
- 로그 보관 데이터
- 캐시가 아닌 중요한 영구 데이터

---

## Velero와 CSI Snapshot의 차이

```text
Velero
  ├── Kubernetes 오브젝트 백업
  ├── 오브젝트 스토리지에 백업 메타데이터 저장
  └── 필요 시 볼륨 백업/복구 연동

CSI Snapshot
  ├── 스토리지 레벨 포인트인타임 복사
  ├── 빠른 복구 가능
  └── 특정 CSI 드라이버가 지원해야 함
```

---

## 어떤 상황에 무엇을 쓰나?

| 상황 | 권장 방식 |
|------|-----------|
| 네임스페이스 전체를 다른 클러스터로 옮기고 싶다 | Velero |
| PVC 데이터만 빠르게 롤백하고 싶다 | CSI Snapshot |
| 앱 리소스와 데이터 모두 같이 살리고 싶다 | Velero + CSI Snapshot |
| 특정 시점 장애 이전 상태로 복구하고 싶다 | CSI Snapshot + Velero |

---

## 백업 전략의 3단계

### 1. 오브젝트 백업
- YAML 정의 복구 가능성 확보
- 빠른 재배포 기반

### 2. 데이터 백업
- PV 데이터 보존
- Snapshot 또는 파일 레벨 백업

### 3. 복구 검증
- 백업이 아니라 **복구 테스트**가 완료되어야 함
- 정기적으로 테스트 클러스터에서 복구 리허설 수행

---

## RPO / RTO

| 용어 | 의미 | 백업 설계에 미치는 영향 |
|------|------|------------------------|
| RPO | 허용 가능한 데이터 손실 시점 | 스냅샷/백업 주기 결정 |
| RTO | 복구 완료까지 허용 시간 | 스냅샷 vs 전체 복원 선택 |

- RPO가 짧을수록 스냅샷 주기를 짧게 잡아야 합니다.
- RTO가 짧을수록 빠른 스냅샷 복구가 유리합니다.

---

## 권장 구조

- **Velero**: 리소스/네임스페이스/클러스터 구성 백업
- **CSI Snapshot**: 중요한 PVC의 데이터 포인트인타임 백업
- **오브젝트 스토리지**: Velero 백업 저장소
- **검증 환경**: 정기 복구 테스트용 별도 클러스터

---

## 핵심 결론

Velero만으로도 상당 부분 복구할 수 있지만, 데이터 무결성과 복구 속도를 높이려면 **CSI Snapshot을 함께 쓰는 것이 가장 실무적**입니다.
